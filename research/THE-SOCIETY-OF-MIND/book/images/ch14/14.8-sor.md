# Figure 14-8 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [14-8.md](14-8.md) — *One-way pipeline view of perception.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This figure is presented by Minsky in order to be rejected. The
analysis below therefore asks two questions: (i) where does the
plan resemble the rejected model, and (ii) where has it already
moved beyond it.

---

## The ideas Figure 14-8 actually carries

1. **A strict left-to-right pipeline.** Each stage's output is the
   next stage's only input.
2. **No back-channel.** Later stages cannot reach earlier ones.
3. **Single-source inputs.** Each stage has exactly one upstream
   producer.
4. **Discrete, named stages.** World, Sensation, Perception,
   Recognition, Cognition are distinct boxes.
5. **An implicit claim of sufficiency.** The diagram suggests this
   chain is enough to account for seeing.
6. **A diagram offered as a foil.** Its very crispness invites the
   reader to feel where it fails.

---

## What the implementation already absorbs

### 1 — A named, ordered phase chain

[`02-workflow-design.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/02-workflow-design.md)
and
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
define the workflow chain
`guard → normalize → guardrail → perceive → activate → deliberate →
criticize → censor → settle → act → remember → report`. This is a
linear sequence of named stages, exactly the shape Figure 14-8
draws.

### 2 — A normalised stimulus as the "Sensation" boundary

`state/runs/<run_id>/stimulus.json`
([`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md))
is the single artefact that crosses the boundary from World to the
rest of the runtime. The figure's `World → Sensation` arrow has a
direct operational counterpart.

### 3 — Append-only state by phase

`state/mind/issues/<n>/` separates `percepts.jsonl`,
`activation.jsonl`, `signals.jsonl`, `objections.jsonl`,
`candidate-actions.jsonl`, `final.md`. Each phase writes its own
file. Ordering is preserved by the workflow, not assumed.

## What the implementation does not yet take into account

### A — The phase chain looks like Figure 14-8 from outside

The pipeline mapping table in
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
is, on its face, a one-way pipeline. The correction Minsky draws
in [14-9.md](14-9.md) — perception fed by expectation, not just by
sensation — is present in the plan (K-line retrieval, frame
selection, prior-decision resonance) but is *not visible in the
phase diagram itself*. A new reader of
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
would see the rejected figure, not the corrected one.

### B — No back-channel from later phases to earlier ones within a
run

Within a single run, `criticize` cannot ask `perceive` to look
again. Objections from critics are filed, but the only way to
"return to perception" is to wait for the next stimulus. Some
forms of revision are not currently expressible inside a single
cycle.

### C — Multi-source inputs are not documented as such

`perceive` reads `stimulus.json` *and* `nemes/*`; `activate` reads
`frames/*` *and* `memory/klines/**` *and* the previous step's
percepts. Each phase already has multiple inputs, but the pipeline
table presents them as a single arrow. The fact that the runtime
is already not Figure 14-8 is not surfaced in its own diagrams.

### D — Sufficiency is not claimed but also not denied

The plan does not claim that its phase chain is *all* that
cognition needs, and it does not deny it either. Figure 14-8's
implicit-sufficiency claim has no explicit refutation in the
plan; the plan simply adds more phases.

## Implication for the plan (no changes proposed here)

Figure 14-8 is the picture of a runtime the plan should not become
even though, on the surface, the plan resembles it. The corrective
material lives in [14-9.md](14-9.md) and in this plan's
expectation-bearing structures (K-lines, frames, prior-decision
resonator) — but those structures do not appear in the pipeline
diagram itself. The gaps above are recorded as analysis, not as a
change request. Any move to close them — for example, a revised
pipeline diagram that draws expectation-arrows back into
`perceive`, or a documented intra-run *re-perceive* path — would
touch the runtime pipeline and activation documents, and so falls
under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
