# Figure 10-2 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [10-2.md](10-2.md) — *Piaget's eggs and cups, AFTER.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

Figure 10-2 is the diagnostic panel: the same scene as 10-1 disturbed
in a single way, and the divergence in verdicts that follows. This
analysis records how the implementation handles disturbances and
divergent judgements.

---

## The ideas Figure 10-2 actually carries

1. **A single controlled disturbance.** Only one variable — spatial
   extent — has changed between 10-1 and 10-2.
2. **Two stable population responses.** Subjects do not produce a
   spread of opinions; they fall into two named groups (younger,
   older) with sharply different verdicts.
3. **The same data licenses opposite conclusions.** *More* and *same*
   are both defensible given a single salient cue.
4. **Appearance can override history.** The younger subject ignores
   the fact that no eggs were added or removed.
5. **History can override appearance.** The older subject ignores the
   visual change because the identity of the eggs is preserved.
6. **A verdict is reasoning made visible.** The "because" clause is
   the data Piaget cares about; it exposes which agent is dominant.
7. **Age is shorthand for architecture.** The figure attributes the
   change in verdict to a change in internal structure, not to more
   information.

## What the implementation already absorbs

### 1 — Controlled disturbances as triggers

A workflow run is triggered by a specific event (push, comment, label)
that names exactly what has changed; the `guard` and `normalize`
phases
([`02-workflow-design.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/02-workflow-design.md))
isolate that change. Each run is, by construction, *one disturbance
applied to a known scene*.

### 2 — Divergent verdicts captured side by side

[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
runs `deliberate → criticize → censor` and records every verdict in
`signals.jsonl` before `settle` produces a single decision. Two
critics that disagree both appear in the record; neither is
suppressed by the other.

### 3 — Reasoning made visible

The critic and censor signal schemas in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md)
include a `summary` field — the "because" of a verdict. The
settlement step preserves these summaries in the run's `report.md`.

## What the implementation does not yet take into account

### A — No mechanism for "appearance overrides history" (or the reverse)

The figure's central claim is that *which agent wins* is a function
of the agency's internal priorities. The plan's `settle` step
([`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md))
applies censors and produces a settlement, but it does not implement
any *priority ordering* between competing critic verdicts. Two
critics that disagree on direction produce a blocked settlement, not
a chosen verdict.

### B — Two stable response populations have no representation

Figure 10-2's "typical 5-year-old" and "typical 7-year-old" are
*modes* of one cognitive system at different stages. The plan does
not model modes: the same workflow runs the same agency catalogue
every time. There is no `mode:` field on the workflow, no
*"developmental stage"* concept, and no way to ask *"what would the
older configuration have decided here?"*

### C — Same scene, different architecture is unsupported

The figure compares verdicts across architectures while holding the
scene constant. The plan can compare verdicts across *runs* (because
state is persisted) but not across *architectures* of itself on the
*same* input, because the agency catalogue is global per repository
and cannot be swapped per run.

### D — Identity-of-the-item as a cue

The older child's argument depends on *"the same eggs"*. The plan
has no critic whose competence is *"are the entities under
consideration identical to the entities considered at the baseline?"*
There is no operational analogue for object-identity reasoning across
runs.

## Implication for the plan (no changes proposed here)

Figure 10-2 is where the chapter's conflict appears. The plan records
conflicts but does not yet resolve them by priority, does not
represent the cognitive *modes* whose existence the figure presumes,
and does not have an object-identity critic. Any move to add a
priority ordering between critics, a mode/stage selector, or an
identity-conservation critic would touch the manifest schema in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the runtime pipeline in
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
and the safety model in
[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
