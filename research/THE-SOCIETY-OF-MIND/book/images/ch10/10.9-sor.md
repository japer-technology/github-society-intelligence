# Figure 10-9 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [10-9.md](10-9.md) — *Building a detour around an old system.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

Figure 10-9 is Minsky's safe-growth method: a new layer is built
*beside* the old one and only allowed to take over once it has been
exercised. This analysis records what the implementation has and
does not have for *parallel, exercised, eventually-intercepting*
growth.

---

## The ideas Figure 10-9 actually carries

1. **New competence is built in parallel, not by editing.** The
   original layer is not modified.
2. **The two layers coexist.** Both are live; both are reachable.
3. **The new layer is exercised before it is trusted.** Stage two
   shows arrows running between old and new — traffic.
4. **Eventual interception is the goal.** Stage three shows the new
   layer above the old, on the path.
5. **Three stages, not two.** *Initial → coexistence → interception*
   is an explicit sequence.
6. **The old layer is preserved indefinitely.** Even in stage three,
   the original box is still drawn.
7. **Growth is *additive*, not *destructive*.** No box is removed
   between stages.

---

## What the implementation already absorbs

### 1 — Branch-based parallel work

Every settlement runs on a branch
([`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md)
default and danger-zone tiers). New competence (a candidate change)
exists *beside* `main` before it is allowed to intercept it. This
is the operational form of *"build a detour beside the old path."*

### 2 — The exercise step is named

[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md)
describes a branch lifecycle: *created → modified, validated,
summarised → merged*. The "validated" step is the figure's
*exercise* step — the new layer is tested against criticism before
adoption.

### 3 — Old and new artefacts coexist via Git

Git history retains the previous state even after merge. Stage
three's "old box still drawn" is preserved by the commit graph.

---

## What the implementation does not yet take into account

### A — The detour pattern is *not* a documented mode for agencies

The branch lifecycle applies to *content*, not to *agencies*. The
plan has no documented procedure for *"introduce a new agency that
runs in parallel to an existing one for N runs, then is promoted
above it."* A new agency is added by writing its manifest and
becomes active immediately — there is no shadow mode.

### B — No "shadow mode" for critics

Figure 10-9 stage two shows the new layer *running but not in
charge*. The plan's critics either run or do not run; they do not
have an *advisory-only* state in which their verdicts are recorded
but not allowed to block a settlement. Such a state is the
operational analogue of *"exercising the new layer."*

### C — No three-stage promotion record

The decisions memory under
`THE-SOCIETY-OF-REPO/06-memory/decisions/` records adopted
decisions but does not record *"agency X was introduced at run 100,
ran in shadow until run 240, was promoted at run 241."* The
promotion arc the figure depends on has no audit shape.

### D — "Old preserved indefinitely" is policy, not enforcement

Git preserves history, but the plan's agency catalogue can have
manifests deleted at any time
([`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md)
does not constrain deletion). The figure's principle that *the old
layer is never removed* is not encoded as a constraint on agencies.

### E — Interception itself is not a primitive

The figure's stage three has the new layer *on the path* of input
between source and the old layer. The plan has no
`intercepts: <other-agency-id>` field, no routing rule of the form
*"signals destined for X now reach Y first."*

## Implication for the plan (no changes proposed here)

Figure 10-9 is the figure that makes Papert's principle *operational*:
not just *insert a layer*, but *grow it beside the old one and let
it earn promotion*. The plan does this convincingly for *content*
(via branches) but does not yet do it for *agencies* (no shadow
mode, no promotion record, no interception primitive). Any move to
add a shadow / advisory mode for critics, a promotion-arc record, or
an `intercepts:` field would touch
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md),
and
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
