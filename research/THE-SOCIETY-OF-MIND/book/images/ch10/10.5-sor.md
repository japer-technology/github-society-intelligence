# Figure 10-5 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [10-5.md](10-5.md) — *Priority ordering of MORE's children.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

Figure 10-5 is the chapter's first proposed conflict-resolution
mechanism: a fixed fall-through priority list. This analysis records
what the implementation already has at the place where priorities
would live and what it does not yet name.

---

## The ideas Figure 10-5 actually carries

1. **Conflict between sub-agents demands an explicit resolver.**
   Without a rule, the agency does not produce a verdict at all.
2. **Priority as the simplest possible resolver.** A linear ordering
   with fall-through.
3. **The resolver is part of the parent.** *MORE*'s job is not to
   judge but to arbitrate.
4. **"If aroused" is a precondition, not a guarantee.** A higher
   child can decline to fire and pass control downward.
5. **A wrong priority order produces wrong verdicts reliably.** The
   younger child's error is not random — it follows from the
   ordering.
6. **The order is data, not code.** It can be changed without
   replacing any of the children.
7. **Fall-through ends at a default.** *CONFINED* is the catch-all
   that guarantees a verdict.

---

## What the implementation already absorbs

### 1 — A parent layer exists between leaves and the apex

The pipeline runs `deliberate → criticize → censor → settle`
([`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)).
The `settle` step is the *arbiter* between competing critic
verdicts — the structural place a priority resolver would live.

### 2 — Decline-to-fire is supported

A critic that does not match its scope simply does not produce a
signal; the pipeline continues. This matches the figure's *"if X is
not aroused, try Y"* semantics at the per-critic level.

### 3 — A default verdict path

The settlement schema in
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
produces a settlement even when no critic fires; the run defaults
to *abstain* or *no-action*. Fall-through to a default is present
in spirit.

---

## What the implementation does not yet take into account

### A — No priority ordering between critics

This is the central gap. There is no `priority:` field on a critic
manifest in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
and `settle.ts` has no rule of the form *"if `critic.evidence` is
aroused, let it decide; otherwise try `critic.consistency`;
otherwise `critic.taste`"*. Conflicting critics either both block or
are resolved by censor logic — never by an ordered list.

### B — Resolver-as-data has no schema

The figure's important property is that the priority list is *data*
that can be edited. The plan has policies
([`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md))
and authority levels (`read`, `draft`, `propose`, `act`, `govern`,
`human`), but no editable priority document under
`.forgejo-society/policies/` that says *"for this agency, this is
the resolver order."*

### C — "Wrong-by-design verdict" is invisible

Because no priority order is recorded, a settlement that was decided
by *the second-priority critic firing after the first declined*
cannot be reported as such. The audit trail in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
captures *which* critics fired but not *who deferred to whom*.

### D — No coupling between resolver and parent agency

Figure 10-5 attaches the priority list to *MORE*, not to the
runtime. The plan's `settle` is global per workflow. There is no
*per-agency* arbiter, and consequently no place to attach an
agency-local priority list. This is the same gap as 10-2's "agencies
as folders, not manifests" gap, viewed from the resolver side.

## Implication for the plan (no changes proposed here)

Figure 10-5 names the missing primitive at the place where the plan
is currently silent: *how do disagreeing critics produce a single
verdict?* The plan has the arbiter (the `settle` step) and the
default path (`abstain`), but no ordering, no per-agency attachment,
and no audit trail of deferral. Any move to add a `priority:` field,
a per-agency arbiter, or a `decided-by-fallthrough` settlement class
would touch
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
and
[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
