# Figure 10-11 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [10-11.md](10-11.md) — *Detour redrawn as a flat hierarchy.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

Figure 10-11 is the *after* picture of the detour: the dashed lines
of [10-10](10-10.md) are gone, the intermediate layer is on the
canonical path, and the diagram looks as if it had always been
this way. This analysis records what the implementation does with
the idea that *strata efface their own history*.

---

## The ideas Figure 10-11 actually carries

1. **A mature hierarchy looks designed.** Nothing in the figure
   suggests it was built by detour.
2. **The old wiring is gone from the picture.** Deprecation has
   completed; the dashed lines are not even drawn.
3. **The intermediate layer is now load-bearing.** Without
   *SPATIAL* and *CONFINED*, there is no path from leaves to
   *MORE*.
4. **The figure presupposes the construction record.** Only because
   we saw 10-9 and 10-10 do we know this layer was once optional.
5. **Three levels, four leaves.** A specific, finite, addressable
   structure.
6. **"Adopted" is a state distinct from "introduced."** Between
   10-10 and 10-11, the new layer has been *adopted*.
7. **The result is reusable as input to the next round of growth.**
   What is now stable is what the next detour will be built
   around.

---

## What the implementation already absorbs

### 1 — A merged branch erases the dashed lines

After merge, the working tree looks designed. Git history still
records the construction, but the live state — the file system the
runtime sees — shows only the *after* picture. This matches the
figure exactly.

### 2 — Adoption is recorded in the decisions memory

`THE-SOCIETY-OF-REPO/06-memory/decisions/` is the place where
"adopted" lives. A decision document is the operational form of
*"this layer is now part of the canonical structure."*

### 3 — Stable structure is the input to further growth

The bootstrap checklist
([`10-bootstrap-checklist.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/10-bootstrap-checklist.md))
sequences items in the order they become reusable for the next item.
The figure's *"adopted → input to next round"* arc is honoured by
the checklist's structure.

---

## What the implementation does not yet take into account

### A — Adoption is not a distinct state on an agency

A new agency is either in the manifest catalogue or it is not.
There is no `adoption_status: introduced | shadow | adopted` field
in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md).
The figure's distinction between "introduced" (10-10) and "adopted"
(10-11) cannot be marked on a manifest.

### B — The construction record is implicit only

Figure 10-11 is meaningful *because* 10-9 and 10-10 preceded it.
The plan keeps construction records in Git history but does not
weave them into the manifest itself. A reader of an agency manifest
cannot tell from the file alone *"this used to be a parallel
shadow of `agency.code.foo`."*

### C — "Looks designed" is not an audited property

The mature hierarchy in 10-11 looks *intentional*. The plan has no
audit pass that says *"the live structure is internally consistent
as if designed from scratch"* — no critic whose job is to evaluate
the *shape* of the agency catalogue rather than the content of any
one settlement.

### D — Reusability for the next round is not declared

The plan does not have a field on an agency manifest saying *"this
is now the stable base on top of which further detours may be
built."* Such a marker would make the *"output of one round becomes
input to the next"* property explicit.

## Implication for the plan (no changes proposed here)

Figure 10-11 is the figure that says *successful growth leaves a
clean diagram*. The plan delivers the clean diagram at merge time
for content, and the decisions memory holds the record of adoption.
What it does not yet have is *adoption as a marked state on the
agency itself*, *construction history woven into manifests*, or a
*"the catalogue looks intentional"* audit. Any move to add an
`adoption_status:` field, a `predecessors:` relation on agency
manifests, or a catalogue-shape audit pass would touch
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and
[`11-mapping-sor-to-implementation.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/11-mapping-sor-to-implementation.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
