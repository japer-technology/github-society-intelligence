# Figure 8-9 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [8-9.md](8-9.md) — *Two hierarchies, two purposes.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 8-9 and checks each
one against the implementation plan.

---

## The ideas Figure 8-9 actually carries

1. **The same entities sort into different hierarchies depending on
   use.** Bird and aeroplane are far apart for the biologist, close
   together for the hunter.
2. **Categories belong to the use, not to the world.** There is no
   privileged hierarchy.
3. **Multiple hierarchies coexist over the same leaves.** Both trees
   are valid; both are addressable.
4. **Switching hierarchies changes what counts as "close".** The
   neighbourhood structure depends on the scheme.
5. **A society needs the ability to *choose* which hierarchy to
   consult for the current task.** Selection is a cognitive act.
6. **New hierarchies can be constructed for new tasks** without
   disturbing existing ones.
7. **Cross-classification is normal, not pathological.** An entity
   genuinely lives in many trees at once.

## What the implementation already absorbs

### 1 — Frames as task-shaped classifiers

`bug.frame.yml`, `feature.frame.yml`, `security-sensitive.frame.yml`,
`code-change.frame.yml`, `question.frame.yml` each impose a
different organisation on the same stimulus
([`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)).
Frames are the closest operational analogue of Figure 8-9's
purpose-shaped hierarchies.

### 2 — Polynemes route the same surface differently

The same path, label, or phrase wakes different agencies depending
on which polyneme matches and which frame is selected. `default_frame:`
on polynemes lets the same symbol prefer a different frame in
different contexts.

### 3 — Analogies as typed structural mappings

`memory/analogies/` holds typed structural mappings between domains
([`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)).
This is a representational hook for the "same thing seen from a
different scheme" idea, even though the first-ship catalogue is empty.

### 4 — Per-use settlement, not per-world settlement

The settlement schema records the *governing frame* and the
*analogies used* per stimulus
([`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md)).
The resolution is tied to the task at hand, not to a single
canonical world model.

## What the implementation does not yet take into account

### A — No explicit *classification scheme* primitive

Hierarchies live implicitly in frame catalogues and family folders.
There is no schema declaring "this is the biologist's tree, that is
the hunter's tree." A reviewer cannot list the society's
classification schemes the way they can list its frames.

### B — Memory records sit in exactly one taxonomy

Every K-line lives under one `memory/klines/<class>/` path. Every
durable record has one storage location. The same K-line cannot be
reached from two equally first-class hierarchies; secondary access
goes through `linked:` references, not through native multi-
classification.

### C — Switching hierarchies is not a runtime lever

There is no operation "for this stimulus, view BIRD and PLANE as
neighbours." Frame selection picks a frame, not a classification
scheme over the leaves. The `kline-retriever` selects K-lines by
cue similarity, not by neighbourhood in a chosen hierarchy.

### D — No "construct a new hierarchy for this task" primitive

Frames are catalogue items added through the `self-modification`
flow. There is no lighter-weight construct for "spin up a hunter's
tree just for this hunt and discard it" — and no construct,
heavyweight or otherwise, for adding a new tree at all separate from
adding a new frame.

### E — Cross-classification is treated as link, not as native presence

`linked: [ kind: analogous_to ]` represents one record's relation to
another. It does not represent one record's *equal membership* in
two hierarchies. Figure 8-9 treats BIRD as genuinely living in both
the ALIVE/ANIMAL tree and the AIR tree; the plan would represent
this asymmetrically.

### F — Hierarchies are static in the catalogue

Frames in the first-ship catalogue are fixed. There is no per-run
discovery mechanism that says "the current stimulus suggests a new
classification of these K-lines; entertain it for this run."

## Implication for the plan (no changes proposed here)

Figure 8-9 is the figure that denies any single privileged
classification. The plan has *task-shaped* selection (frames) but
not *taxonomy-shaped* selection over the leaves of memory. Closing
the gaps would touch the frame schema and analogy catalogue in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
the memory layout in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and the SOR mappings for hierarchy and relational memory in
[`11-mapping-sor-to-implementation.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/11-mapping-sor-to-implementation.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
