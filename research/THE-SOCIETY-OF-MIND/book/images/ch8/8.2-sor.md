# Figure 8-2 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [8-2.md](8-2.md) — *Two K-lines firing into the same field.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 8-2 and checks each
one against the implementation plan.

---

## The ideas Figure 8-2 actually carries

1. **K-lines coexist.** More than one K-line can be active in the
   same field at the same time.
2. **An old memory can wake mid-task.** kP fires *after* the Q-pattern
   is already running, not before deliberation begins.
3. **Recognition fires kP — not deliberate retrieval.** Some current
   feature reminds the field of P; the wake is data-driven, not
   commanded.
4. **The two K-lines write into a shared substrate.** Overlap is
   structural: any agent could in principle be in both attached sets.
5. **Each K-line retains its identity.** kQ's contribution and kP's
   contribution remain distinguishable in the field.
6. **The figure stages a conflict question.** Two patterns over the
   same slots set up the policies of Figure 8-3.
7. **Concurrent activation is the normal case, not the exception.**
   The figure presupposes that memory is multiply present, always.

## What the implementation already absorbs

### 1 — Multiple K-lines load per stimulus

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
makes `klines.ts` select up to `memory.max_k_lines_loaded` K-lines and
boost every agency in each chosen snapshot. Multiple K-lines coexist
in the activation pass by construction.

### 2 — Per-K-line bookkeeping survives the run

Each reactivation is recorded in `state/.../activation.jsonl` with
the K-line id ([`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)).
Credit assignment can later attribute outcome credit per K-line
([`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)).
Identity is preserved through to `remember`.

### 3 — Recall happens *inside* current processing

`activate` runs after `perceive`, so K-line reactivation occurs
in the same workflow run as the new stimulus's deliberation. This is
the figure's "old memory wakes in the middle of a new task" rendered
operationally.

---

## What the implementation does not yet take into account

### A — No "recognition mid-deliberation"

K-line retrieval runs *once* in the `activate` phase. Later phases
(`deliberate`, `criticize`, `censor`, `settle`) cannot trigger a
fresh K-line wake from a partial cue produced by the deliberation
itself. The figure's kP is fired by recognition *during* problem-
solving; the plan fires all K-lines up front.

### B — The current stimulus is not itself a K-line

In the figure, kQ is just another K-line that happens to be the one
the current task is built on. In the plan, current activation is
constructed fresh from `perceive` + frame + polynemes; it is not a
K-line peer with which a recalled kP could overlap. The "two patterns
in one field" framing has no native expression.

### C — No collision record

When multiple K-lines boost the same agency, weights are combined
(summed or otherwise composed) and no record is written that says
"agency X received +0.4 from kP and +0.3 from kQ." Settlement
preserves final activated weights, not the K-line-by-K-line
contribution. Figure 8-2's structurally interesting moment — two
K-lines touching the same slot — leaves no audit trail.

### D — No notion of contested vs uncontested overlap

The figure distinguishes agents reached by *both* K-lines from
agents reached by only one. The plan has no "intersection set"
concept on the activation pass. Downstream critics see the union,
not the structure.

### E — Reactivation is biased toward similar cues, not toward fertile interference

`restore_when` similarity tends to pick K-lines from the *same kind*
of past episode. The figure's interesting case is when a K-line from
an unrelated past is summoned by some incidental feature — the
fruitful surprise. The plan's selection function does not seek out
that kind of cross-cue recall.

## Implication for the plan (no changes proposed here)

The plan handles the *coexistence* portion of Figure 8-2 cleanly.
What it does not yet model is the figure's distinctive *moment* — a
recalled memory firing while a current pattern already holds the
field — nor the structural information that such overlap creates.
Closing gaps A–D would touch the activation protocol in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
the signal/handoff schemas in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
and the pipeline ordering in
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
