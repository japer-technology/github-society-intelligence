# Figure 8-1 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [8-1.md](8-1.md) — *A K-line attached to active agents.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 8-1 and checks each
one against the implementation plan.

---

## The ideas Figure 8-1 actually carries

1. **A K-line is itself a new agent.** It enters the same field as the
   agents it remembers, not a separate store.
2. **Construction rule: attach to whatever is active right now.** The
   K-line's content is the activation snapshot of an episode.
3. **The cue for construction is the episode, not a description of it.**
   What gets recorded is *which agents were on*, not a summary of the
   problem.
4. **The attached set is heterogeneous.** Agents are drawn from
   anywhere in the field — no respect for family boundaries.
5. **Re-activation is fan-out.** Firing kP excites every attached
   agent in parallel.
6. **The K-line is external to the agents it touches.** The original
   agents need not know they were memorised.
7. **A shared substrate is presupposed.** The "field of agents" is
   the connective tissue that lets a K-line reach in at all.
8. **Granularity is preserved.** kP records *which* agents, with
   identity intact — not merely *that* the problem was solved.

## What the implementation already absorbs

### 1 — Snapshot of activation, with weights

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
and [`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md)
define the K-line schema with `activation_snapshot.active_agencies:
<agent-id>: 0.0`. This is Figure 8-1's "darkened agents" with weights
attached — the per-agent identity is preserved.

### 2 — Construction at the end of a successful episode

The `archivist` in the `remember` phase writes a K-line only when the
settlement outcome was `success` or `partial`
([`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)).
This is the figure's "build a K-line from the agents that were active
while problem P was being solved."

### 3 — Re-activation as fan-out

`klines.ts` in the `activate` phase boosts the activation of every
agency in each chosen K-line's `activation_snapshot` and records the
reactivation per K-line id. This is the figure's right-hand panel:
kP fires, every attached agent lights up.

### 4 — K-lines are durable, governed, addressable

`memory/klines/<class>/<date>-<stimulus_id>.yml` gives every K-line a
stable id and location. The K-line is a thing one can point at.

## What the implementation does not yet take into account

### A — A K-line is a record, not an agent

In the figure, kP is a *new agent* added to the field — addressable
by the same mechanisms as any other agent. In the plan, a K-line is
a YAML record under `memory/klines/`. It has no manifest, no
`activates_on`, no `inhibits`, no tools, no authority line, and no
budget. Only `klines.ts` reads it. Nothing in the runtime can be
written to "wake kP" the way one would wake `agency.code.patch-imaginer`.

### B — Attachment is one-shot, not incremental

The figure's K-line is built from the agents that *were* active
during the episode — implicitly the union as the episode progressed.
The plan writes the snapshot once, at `remember`, from the final
settlement. There is no notion of incremental attachment as more
agents become active during deliberation, nor of an early-vs-late
distinction inside the snapshot.

### C — Reactivation similarity is matched on cue, not on activation overlap

`restore_when` is built from stimulus fields (terms, labels, paths,
symbols), not from "this stimulus's current activation overlaps kP's
attached set." Two stimuli that activate the same agents but use
different labels will not retrieve each other's K-lines. The figure's
matching criterion (fire kP when its agents look like the right
constellation) is approximated indirectly via cue similarity.

### D — No cascading from K-lines back into other K-lines

A fired K-line excites agencies; it does not excite other K-lines.
The plan has no mechanism by which kP, on firing, raises the
activation of agencies that then match a *further* K-line's
`restore_when`. Figure 8-1's premise that K-lines live in the same
field as agents would naturally permit such cascades.

## Implication for the plan (no changes proposed here)

Figure 8-1 is the figure that defines the K-line. The plan honours
the *outcome* of that definition (a durable, reusable activation
snapshot) but not the *form* (a new agent placed into the same
field). Closing gap A — making a K-line a first-class agent kind —
would touch the manifest schema in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the activation protocol in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
and the identity scopes in `THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`,
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
