# Figure 8-8 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [8-8.md](8-8.md) — *A K-line tree for "Jack flies a kite".*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 8-8 and checks each
one against the implementation plan.

---

## The ideas Figure 8-8 actually carries

1. **Memory naturally organises as a hierarchy of recognisable
   kinds.** EVENT → actor / verb / object → attributes → raw agents.
2. **The level-band rule (Figure 8-6) produces the layered shape.**
   Each layer captures a category that is independently meaningful.
3. **Leaves are shared across parents.** The same low-level agent
   can be reached from many places in the tree.
4. **Hierarchical memory makes partial recall meaningful.** Touching
   JACK lights the actor cluster without summoning the whole event.
5. **The tidy hierarchy is an ideal, not a guarantee.** Real
   knowledge-trees get tangled by shortcuts and exceptions
   (Minsky's warning).
6. **Hierarchy is the structure that lets the figure's tree be
   walked.** Recall and reasoning depend on traversal, not just on
   matching.
7. **Each layer can decay or change independently.** Attributes can
   shift without rewriting the whole event memory.

## What the implementation already absorbs

### 1 — Hierarchy of *summaries*

`agencies/assembly/summary-tier-1` and `summary-tier-2` compress raw
signals into per-family summaries and then into a settlement-ready
brief ([`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md)).
This is hierarchy applied to *outputs* — the protocol mapped in
[`11-mapping-sor-to-implementation.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/11-mapping-sor-to-implementation.md)
as `02-protocols/13-hierarchy-and-summaries.md`.

### 2 — Typed relational DAG on durable records

`linked:` with kinds (`derived_from`, `cites`, `reinforces`, …)
([`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md))
gives every record a place in a typed graph. The substrate for tree
construction exists.

### 3 — Coarse class hierarchy on K-lines

`memory/klines/<class>/` groups K-lines by class. This is one layer
of hierarchy — enough to organise, not enough to traverse.

### 4 — Append-only with supersession

Per-layer independent change has a partial analogue: a record can be
superseded without rewriting its neighbours.

## What the implementation does not yet take into account

### A — Hierarchy is over summaries, not over K-lines themselves

The plan's hierarchical primitive lives in `agencies/assembly/`. It
compresses outputs in one run. There is no `klines/event/...` →
`klines/actor/...` → `klines/attribute/...` chain that the runtime
walks when recalling a composed memory.

### B — Partial activation of a sub-tree is not supported

K-lines are loaded as wholes by `klines.ts`. Figure 8-8's "touch
JACK, light the actor cluster" — partial entry into the tree
without instantiating the whole event — has no operational form.

### C — Class folders are flat

`memory/klines/<class>/` (e.g. `code-change/`, `security/`,
`question/`, `self-modification/`) is one level deep. The deeper
category structure of Figure 8-8 — EVENT/ACTOR/VERB/OBJECT/ATTRIBUTE
— has no representation in the directory layout or in any K-line
field.

### D — Shared leaves are not enforced

Two K-lines referring to the same low-level concept hold independent
records. There is no canonical leaf store from which multiple parents
draw. The "same agent reachable from many parents" property of the
figure is, at best, expressed via `linked:` references that the
runtime does not currently exploit on recall.

### E — Independent per-layer decay is absent

`decay_score` is per-K-line. There is no layer-aware decay that, for
example, refreshes attribute-layer K-lines while letting event-layer
K-lines age, or vice versa. Figure 8-8's "each layer can change
independently" has no scheduler analogue.

### F — Tangled-tree warning is unrepresented

Minsky's caution — real knowledge-trees grow shortcuts and
exceptions — has no operational signal. The plan treats memory as
append-only with typed links, but does not run a periodic
re-organisation pass that detects accumulated tangling against a
declared tree shape.

### G — Traversal is not a runtime operation

The relational index in [`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
is built lazily on first read; nothing in the deliberation phases
walks it to *reason over* the tree. K-lines contribute to activation,
not to traversal-based recall.

## Implication for the plan (no changes proposed here)

Figure 8-8 turns the *graph* of Figure 8-7 into a *tree* — a
traversable, layered structure that supports partial recall and
hierarchical reasoning. The plan has the substrate (typed links,
summary tiers) but not the operational tree. Closing the gaps would
touch the K-line schema and reactivation loop in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
the memory layout and decay in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and the hierarchy protocol mapping in
[`11-mapping-sor-to-implementation.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/11-mapping-sor-to-implementation.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
