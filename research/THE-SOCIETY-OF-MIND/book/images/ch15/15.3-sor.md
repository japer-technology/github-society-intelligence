# Figure 15-3 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [15-3.md](15-3.md) — *anatomy of memory: layered
memory machinery inside a typical agency.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 15-3 and checks each
one against the implementation plan. The aim is to mark, plainly, what
the plan already absorbs and what it does not yet name.

---

## The ideas Figure 15-3 actually carries

Pulled from the figure caption and Minsky's surrounding text in §15.8
*anatomy of memory*:

1. **Memory belongs inside an agency, not only beside it.** Every
   substantial agency carries its own memory plant; memory is not a
   single external store the agency reads from.
2. **Layers with different speeds.** Micromemory-units (fast,
   temporary), short-term units (slower, holds place across
   interruption), long-term storage (durable). Each tier has its own
   write rules.
3. **Each tier protects the tier beneath it.** A short-term unit
   shields a micromemory from being overwritten before its content is
   used or promoted.
4. **A micromemory is a temporary K-line.** It can save or restore the
   state of *many* agents at once — it is not a single value but a
   reactivation pattern across an internal population.
5. **Save/restore is a primitive operation.** The figure presupposes
   that a unit can snapshot the active configuration of its
   neighbours, store it, and re-impose it later.
6. **Outward arrows: long-term memory and neighbouring agencies.** The
   internal memory plant exchanges material with the durable store and
   with peer agencies; it is not sealed.
7. **A memory-controller sits inside the agency.** Flow between tiers
   is managed by a named control block, not by ambient runtime rules.
8. **The controller itself contains memory.** The recursive answer to
   "how does the controlling machinery learn to control" is that the
   controller has its own miniature memory plant — memory all the way
   down.

These eight items are what the rest of this file tests against the
implementation plan.

## What the implementation already absorbs

### 1 — Multiple lifetimes, multiple write rules

[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
defines exactly three trees with distinct lifetimes and write rules:
`state/` (per-run, runtime-only), `workspace/` (short-term, swept after
settlement), `memory/` (durable, append-only, `archivist` only).
Promotion is one-way. This is the *spirit* of Figure 15-3's layered
plant — fast/transient on the inside, slow/durable on the outside,
with a guarded promotion path between them.

### 2 — K-lines as the reactivation primitive

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
and the K-line schema in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md)
treat a K-line as a saved pattern of which agencies were active and
why, retrievable by `agency.memory.kline-retriever`. The figure's
"micromemory-unit is a temporary K-line that saves the state of many
agents at once" is the same shape, scaled up to the run as a whole.

### 3 — A named promoter, not ambient persistence

`archivist` is the only writer to `memory/`
([`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)).
That answers Figure 15-3's insistence that there *is* a controller
moving material between tiers, rather than tiers diffusing into each
other on their own.

### 4 — Short-term as a real, swept tier

`workspace/` exists precisely as the figure's middle layer: it holds
the in-flight situation, briefings, and forming settlements; it is
mutable in place; it is swept by `report` into `memory/decisions/`.
This is "short-term memory-units keep place across interruption" in
operational form.

### 5 — Per-issue scratch as fast, disposable state

`state/mind/issues/<n>/` (percepts, activation, signals, blackboard,
candidate K-line) is the closest thing the plan has to a micromemory
tier: per-stimulus, fast, overwritten-by-run, with a single candidate
`kline.yml` that the `archivist` may promote. The figure's "innermost
fast layer feeding an outer slower one" is honoured at the *run*
scale.

## What the implementation does not yet take into account

These are the parts of Figure 15-3 that the plan currently leaves
implicit, partial, or simply absent. None are urgent; all are real.

### A — Memory is run-scoped, not agency-scoped

The figure's central claim is that *each substantial agency* carries
its own three-tier plant. The plan's three trees are **global to the
run**: there is one `state/`, one `workspace/`, one `memory/`. There
is no `agencies/code/patch-imaginer/state/` or
`agencies/code/patch-imaginer/workspace/`. An individual agency
cannot, today, hold a private fast store that survives between
activations within a run but does not leak to peers. "Memory inside
the agency" is not a representable shape.

### B — No save/restore primitive across a named set of agents

A micromemory-unit, in the figure, can snapshot and re-impose the
state of *many* agents at once. The plan can record *which* agencies
were active (`activation.jsonl`, `signals.jsonl`) and reuse that
pattern via `kline-retriever`, but it has no operation that means
"freeze the current activation vector of agencies X, Y, Z and re-fire
it later, atomically." K-line reuse today is a *suggestion* injected
during `activate`, not a restore of an internal cognitive
configuration.

### C — No agency-local micromemory tier

The plan has fast (`state/`) and slow (`memory/`) but no tier *below*
`state/` that is shorter-lived than a run — e.g. a per-phase or
per-activation scratch that an agency uses while it deliberates and
then discards before the next agency runs. The figure's innermost
ring (the bank of micromemories) has no operational analogue. All
intra-run scratch currently sits in process memory inside the runtime
and is not separately named, schema'd, or addressable.

### D — Tiers do not "protect" each other; promotion is monotonic

In the figure, a short-term unit's job is partly *defensive*: it
keeps an outer tier from overwriting an inner one before the inner
one has been read. The plan's tiers are append-only and promote in
one direction, but they do not model interference between tiers.
There is no field that says "this micromemory must be latched into
short-term before the next activation wave runs." Race conditions
between activation waves are handled, if at all, by the workflow
phase order — not by a memory-tier guard.

### E — No internal memory-controller per agency

`archivist` is one agency in the `memory` family that writes the
durable store. The figure's controller is *inside every substantial
agency* and routes that agency's own tiers. The plan has no
`agency.code.patch-imaginer.memory-controller` companion, no
per-family memory-controller, and no schema field on an agency
manifest that declares "this is my memory-controller." Flow between
tiers is a property of the runtime, not of the agency.

### F — The recursive case: controller-of-controller has no memory plant

Figure 15-3's deepest point is that the memory-controller itself
contains a memory-system, so the controlling machinery can learn.
The plan's `agencies/meta-admin/` family
([`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md))
contains `differentiation-broker` and report-writers, but none of
them have a private layered memory of *their own* control decisions.
Meta-admin units write into the same global `memory/` as everyone
else, with no inner tier they can keep private and refine. The
"memory all the way down" property is asserted nowhere in the
schema.

### G — Lateral arrows: agency-to-agency memory exchange

The figure shows arrows from an agency's memory plant *across to
neighbouring agencies*. In the plan, agencies communicate via signals
and handoffs
([`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md)),
not by reading each other's memory. There is no "agency A reads
agency B's short-term store" channel, and no policy in
[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md)
governing such a read. Inter-agency memory exchange is currently
mediated entirely by the global `workspace/` blackboard.

### H — Granularity vocabulary: *memory* vs *memory-unit* vs *memory-tier*

The plan uses *memory* as both the tree (`memory/`) and the family
(`agencies/memory/`). The figure depends on three distinct words:
the *plant* (the whole thing inside an agency), a *tier* (fast,
short, long), and a *unit* (one micromemory or one short-term cell).
Without this vocabulary, gaps A–G cannot be described precisely in
any current implementation document — they collapse into
"`memory/`" or "K-line", neither of which fits.

## Implication for the plan (no changes proposed here)

Figure 15-3 is the figure that says *every substantial mind-part owns
its own miniature memory-economy*. The implementation plan honours the
*outcome* of that figure (named tiers, a single durable writer,
K-lines as reactivation patterns) at the **run** scale, but does not
yet honour the *locality* of the figure (each agency carries its own
plant, with its own controller, recursively) at the **agency** scale.

The gaps above are recorded here as analysis, not as a change
request. Any move to close them — for example, a per-agency
`state/`/`workspace/` subtree, a `memory_controller:` field on the
agency manifest schema, a `save_activation` / `restore_activation`
primitive over a named agent set, or a recursive memory-plant
declaration on meta-admin units — would touch the manifest schema in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the memory layout in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
the K-line schema in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
and the memory protocol in
`THE-SOCIETY-OF-REPO/02-protocols/06-memory.md`, and so falls under
the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
