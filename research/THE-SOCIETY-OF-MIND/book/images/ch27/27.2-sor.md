# Figure 27-2 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [27-2.md](27-2.md) — *Suppressor on the path from
present state to an undesired action.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 27-2 and checks each
against the implementation plan. The aim is to mark, plainly, what the
plan already absorbs and what it does not yet name.

---

## The ideas Figure 27-2 actually carries

1. **A suppressor sits between present state and action.** It is a
   *late* intercept, on the last arrow before the boundary.
2. **The suppressor is a separate agent.** It is drawn as its own
   box, with its own arrow into the path; it is not a property of
   the state.
3. **The bad thought has to happen first.** The figure depicts
   *Present state* and only then the suppressor cutting the link to
   *Undesirable Action*. Catching requires that the candidate already
   exist.
4. **The cut is on the link, not the box.** The state is allowed to
   form; only the consequence is blocked.
5. **Recognition is the suppressor's whole job.** It does not propose
   alternatives. It says *not this*, and stops.
6. **One suppressor per learned mistake.** Each suppressor is paired
   with a previously seen present-state-to-action edge.
7. **Suppression is costly.** Time and effort are spent forming the
   thought before it is discarded.
8. **A suppressor implies prior experience.** Something must have
   gone wrong, once, for the agent to know which state-to-action
   edge to watch.

## What the implementation already absorbs

### 1 — Suppressors as a named, distinct layer

[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md)
defines suppressors as their own row in the layered safety model,
with their own firing point (*on each candidate output*) and their
own outcome vocabulary (`stop_and_comment`, `open_pr_only`,
`request_human`). The figure's "separate box, separate arrow"
treatment is honoured in the schema.

### 2 — Suppression on the last arrow

Suppressors run in the `act` phase, on candidate outputs (diffs,
comments, branch contents), before they cross an effect boundary.
This is exactly the position the figure assigns them — between the
formed candidate and the action.

### 3 — One suppressor per learned edge

The first-ship catalogue
(`suppressor.workflow-diff-keywords`, `suppressor.soul-file-diff`,
`suppressor.high-entropy-string` in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md))
is one-suppressor-per-known-mistake-shape, which matches Minsky's
"one per edge" reading.

### 4 — Recognition without proposing alternatives

A suppressor's verdict set is `stop_and_comment | open_pr_only |
request_human`. None of these *replace* the candidate with a
corrected version; they only block, divert, or escalate it. That is
the figure's "say *not this* and stop" exactly.

### 5 — The cut is recorded as a signal

Suppressor activations are written to
`state/runs/<run>/objections.jsonl` per
[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md).
The cost the figure depicts — *the bad thought had to happen* — is
made visible as an audit trail rather than discarded.

## What the implementation does not yet take into account

### A — Suppressors are not learned from past runs

The figure assumes *prior experience* — the suppressor exists
because the agent saw the edge fail once. The plan's suppressors are
authored, not derived: they live in `policies/danger-zones.yml`
suppressor blocks and `censors/*.md` files with `kind: suppressor`,
all of which are written by hand and reviewed under the
`self-modification` frame. Nothing in the runtime inspects
`memory/failure/rejected-candidates/`
([`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md))
to *propose* a new suppressor when the same present-state-to-action
edge recurs.

### B — No representation of "present state" as a recognisable object

The figure makes *Present state* a labelled box that the suppressor
can match on. The plan has no first-class state object that names
the cognitive condition immediately preceding a candidate output. A
suppressor matches on the *candidate* (a diff, a regex, an entropy
score), not on the conditions under which the candidate was formed.
`activates_on` and frames carry some of this load, but only at
entry; there is no `pre_action_state` snapshot a suppressor can read.

### C — Cost is not accounted for

The figure's whole reason for then introducing censors is that
suppressors are *expensive*: thought happened, then was thrown
away. The plan records token, tool-call and wall-clock budgets per
agency
([`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md))
but does not attribute *wasted* budget to the suppressor that
discarded the candidate. There is no field that says "this run spent
N tokens on a deliberation that the X suppressor then voided."
Without that, the figure's argument-by-cost cannot be made empirical.

### D — One suppressor cuts one link

The figure depicts a single suppressor on a single edge. The plan
has no mechanism for a *family* of related suppressors that share
the same recognition core (e.g. several `secret-shaped` suppressors
across different egress paths) and update together when the core
recognition changes. Each suppressor is independent.

### E — Suppressors do not feed forward into earlier phases

The chapter motivates censors as the natural next step: *if you've
had to suppress here repeatedly, learn to censor earlier.* The plan
has no path by which a suppressor that fires often produces a
candidate censor or guardrail, automatically or by ticket. The
relationship between suppressor activations and earlier-phase
hardening is informal.

### F — No notion of "almost suppressed"

The figure is binary — the suppressor either cuts the link or does
not. The plan inherits this. A candidate that passes a suppressor by
a narrow margin (one keyword short of the regex set, entropy just
below threshold) is indistinguishable in `objections.jsonl` from one
that passes comfortably. The data needed to tune thresholds is not
captured.

## Implication for the plan (no changes proposed here)

Figure 27-2 is the figure that argues *intercepting late is the
cheap, obvious place to start, and the wrong place to stop.* The
plan adopts the cheap start — suppressors exist, fire on candidates,
and leave audit traces — but does not yet account for the cost or
the learning that the figure says should drive censors into
existence over time.

Any move to close the gaps above — a `pre_action_state` snapshot, a
`wasted_budget_attributed_to:` field on suppressor verdicts, an
automated proposer that reads `memory/failure/` and drafts new
suppressors or censors, or a near-miss margin recorded with each
activation — would touch the signal schema in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
the suppressor catalogue in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the safety layers in
[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md),
and the memory layout in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
