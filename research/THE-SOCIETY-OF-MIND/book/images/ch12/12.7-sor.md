# Figure 12-7 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [12-7.md](12-7.md) — *Cross-linked bridges between
structures (arch, apertures, legs, columns, hole) and functions
(bridge, passages, supports, door, window).*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

---

## The ideas Figure 12-7 actually carries

1. **Structures are reused across functions.** A single *arch* can
   serve as bridge, passage, support, doorway, window-frame.
2. **Functions are reused across structures.** *Support* may be
   played by an arch, columns, or legs.
3. **The mapping is many-to-many.** The arrows form a web, not a
   tree or a list of pairs.
4. **Power of thought scales with cross-link density.** Many
   bridges, not many nodes, are what make a concept fertile.
5. **A bridge does not need a unique name.** Several arrows in the
   diagram coexist without identifiers; the *set* of arrows is the
   knowledge.
6. **The "etc." matters.** The diagram explicitly admits the
   catalogue is open. Bridges can be added without revising existing
   ones.
7. **Reformulation is a navigation problem.** Solving a new task
   means following structure → function → other structure across the
   bridges.

## What the implementation already absorbs

### 1 — Many-to-many links between records

`linked: [ <record-id>, ... ]` is a list, and any number of records
can link to any other
([`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)).
The graph shape is supported.

### 2 — Analogies as a memory tree

`memory/analogies/<analogy-id>.yml`
([`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md))
holds "cross-domain mappings" — the closest existing home for
structure↔function webs that span more than two nodes.

### 3 — Open-ended catalogues

Frame, polyneme, K-line, and concept catalogues are all open: new
entries can be added without revising existing ones. The plan's
"etc." discipline matches Figure 12-7's.

### 4 — Indexing for traversal

`THE-SOCIETY-OF-REPO/02-protocols/14-relational-memory.md` (cited by
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md))
provides a transient index over relational links so the runtime can
traverse them at activation time. This is the substrate for
*following arrows*.

## What the implementation does not yet take into account

### A — No bipartite *structure ↔ function* graph

Even if many `linked:` edges exist, nothing distinguishes structure
nodes from function nodes. There is no schema that partitions
`memory/concepts/` (structures) from `memory/procedural/` (functions)
as the two sides of a bipartite graph, and no query that asks "given
structure X, what functions does it realise?"

### B — Reformulation is not a runtime phase

The pipeline `guard → normalize → guardrail → perceive → activate →
deliberate → criticize → censor → settle → act → remember → report`
([`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md))
has no phase named *reformulate* — no step whose job is *try this
under a different function and see if the available structures fit*.
The "navigation across bridges" idea has no operational home.

### C — Analogies are entries, not links

`memory/analogies/<analogy-id>.yml` is a *record* with two sides. The
web in Figure 12-7 is *edges*, not nodes. Representing the same web
as several analogy records would multiply files without making the
graph queryable.

### D — Cross-link density is not measured

The plan tracks `reuse_count` and `decay_score` on individual records
([`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md))
but not "how many bridges does this concept participate in." The
metric Figure 12-7 implicitly cares about — *node fertility* — has no
runtime counterpart.

### E — Polynemes excite agencies, not structures

Polynemes ([`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md))
wake agencies. They do not retrieve concepts. A path polyneme firing
*workflow-file* does not pull up the *concept* `workflow-file` and
its function bridges. The cross-realm wakeup happens at the
*executor* level, not the *idea* level.

## Implication for the plan (no changes proposed here)

Figure 12-7 turns the structure↔function bridge into a *web* and
makes the web's density the source of cognitive power. The plan can
*store* such a web (via `linked:` edges across `memory/concepts/`
and `memory/procedural/`) but cannot *partition* it, *measure* it, or
*navigate* it as a deliberate runtime move. Any change to add a
bipartite query layer, a density metric, or a `reformulate` phase
would touch
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and [`11-mapping-sor-to-implementation.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/11-mapping-sor-to-implementation.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
