# Figure 12-11 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [12-11.md](12-11.md) — *Single, leaning, stack, and
fireplace arches — real arches that escape the block-arch uniframe.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

---

## The ideas Figure 12-11 actually carries

1. **A well-formed uniframe still under-covers.** The block-arch
   description is right *and* incomplete for the category *arch*.
2. **The leftover instances are themselves structured.** Each escaped
   arch (single, leaning, stack, fireplace) has its own internal
   description; they are not unparsed exceptions.
3. **Multiple uniframes can co-exist under one name.** *Arch* is
   served by several descriptions, each covering a sub-family.
4. **The set of sub-families is open.** The figure displays four;
   it does not claim there are only four.
5. **Naming pressure exceeds description pressure.** People say
   *arch* about all four shapes before any single description
   covers them.
6. **The uniframe-vs-accumulation choice is local, not global.** A
   concept can be uniframe-shaped in one region and accumulation-
   shaped in the leftovers.
7. **Recognition cost grows.** A learner who knows several arch
   uniframes must choose among them, which is itself work.

## What the implementation already absorbs

### 1 — Multiple frames per concept area

Nothing in [`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
forbids several frames from competing for the same stimulus. The
first-ship catalogue already has overlapping frames (`bug`,
`feature`, `code-change`) which can all partially match a single
event, with `default_frame:` on polynemes biasing the choice.

### 2 — Frame selection by best-match

The `activate` phase ([`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md))
selects frames by signal/label/path/phrase match. Choice among
competing descriptions is already a runtime step.

### 3 — `novel.frame.yml` for the leftover

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
ships a `novel.frame.yml` that fires when "no frame matches strongly"
and forces an analogy pass. That is the closest operational analogue
of "an arch we have no uniframe for."

### 4 — Open catalogue of frames

Frames are files; adding a `leaning-arch.frame.yml` to a hypothetical
arch domain is a normal authoring action, not a schema change.

## What the implementation does not yet take into account

### A — No grouping of frames under a common name

Figure 12-11's central claim is that *arch* names a *family of
frames*. The plan has no schema that groups several frames under one
name. A reader of `frames/` cannot ask *which frames are sub-frames
of arch?* and get an answer; only the first-ship table in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
collates frames, and that table is hand-curated.

### B — No mixed uniframe/accumulation representation

The leftover arches in Figure 12-11 are *enumerable*. The plan
cannot say *concept Y is described by uniframe U for cases that fit,
and otherwise enumerated as members M1, M2, M3*. That hybrid is
Figure 12-12's point; the plan has neither side of it formally.

### C — Recognition cost is not modelled

`budget.max_tool_calls` and `budget.max_wall_clock_s`
([`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md))
limit work, but no metric is kept for *how many frames had to be
considered before one was selected*. Figure 12-11's growing
recognition cost has no operational counter.

### D — The `novel` frame discards, not preserves, the leftover

`novel.frame.yml` forces an analogy pass; it does not record *this
stimulus was an unrecognised member of a known family*. A leaning-
arch entering the system today would be handled as *novel* and lose
its family attribution.

### E — Frames cannot declare *sibling-of*

A frame has no field saying *this frame is a sibling of frame X
under family Y*. Sibling relationships among frames are not
modelled, so the set in Figure 12-11 cannot be expressed even
informally inside `*.frame.yml` files.

## Implication for the plan (no changes proposed here)

Figure 12-11 makes the case that *a concept can outgrow its first
description without becoming an accumulation*. The plan can host
many frames and pick the best match, but it cannot group them under
a shared name or admit *we have several uniframes for one concept
plus a small leftover*. Any change to add a frame-family identifier
(e.g. `frame-family.arch`), sibling-of links between frames, or a
mixed-representation schema for `memory/concepts/` would touch
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and `THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`, and so falls
under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
