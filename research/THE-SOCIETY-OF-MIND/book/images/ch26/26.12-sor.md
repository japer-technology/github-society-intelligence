# Figure 26-12 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [26-12.md](26-12.md) — *Adjective order in an
English noun phrase.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 26-12 and checks each
one against the implementation plan. The aim is to mark, plainly, what
the plan already absorbs and what it does not yet name.

---

## The ideas Figure 26-12 actually carries

Pulled from the figure and Minsky's surrounding text in §26.7
*frames for nouns*:

1. **A noun phrase is itself a frame.** *One of the first three big
   round white hat boxes* is parsed by reading words into a fixed
   schedule of terminals.
2. **The schedule is ordered.** *Selector → determiner → orderer →
   number → size → shape → colour → classifier → noun* is not
   convention; it is the frame's slot sequence.
3. **Out-of-order fillers are rejected, not re-routed.** *Wooden
   three heavy brown big first boxes* fails not because each word
   is wrong but because the order violates the frame.
4. **Several slots can be grouped into one composite filler.** *Big
   round white* are bracketed under the single label POLYNEME
   because they describe one object jointly.
5. **A polyneme can be a *group of adjacent slots*, not only a
   single symbol.** The figure widens the polyneme idea from
   §26.7's earlier use.
6. **Slot identity is by role name, not by position.** The labels
   (*SIZE*, *SHAPE*, *COLOR*) carry the meaning; the cells under
   them only hold the current filler.
7. **Empty slots are permitted.** Most noun phrases skip several of
   these slots and remain well-formed; the frame allows omission as
   well as ordering.

These seven items are what the rest of this file tests against the
implementation plan.

## What the implementation already absorbs

### 1 — Slots are named roles, not positions

The frame schema in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
keys slots by name (`relevant_files`, `proposed_patch`, `tests`,
`risks`). This matches Figure 26-12's reliance on *SIZE* and
*COLOR* as labels rather than as positions in a list.

### 2 — Polynemes already exist as a first-class kind

`path-polynemes.yml`, `label-polynemes.yml`, and
`phrase-polynemes.yml` in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
implement polynemes as activators that wake many partial meanings.
Figure 26-12 is using *polyneme* in the same family, even if it
applies the term to a group of slot fillers rather than to a single
symbol.

### 3 — Optional slots

The frame schema's `required: true|false` per slot lets a frame
declare which terminals must be filled and which may be omitted.
This matches Figure 26-12's tolerance of noun phrases that skip
*colour* or *shape*.

### 4 — Pipeline ordering

The workflow phase chain
(`guard → normalize → guardrail → perceive → activate → deliberate
→ criticize → censor → settle → act → remember → report`) is an
ordered schedule at the *run* scale, matching Figure 26-12's claim
that schedules govern composition.

## What the implementation does not yet take into account

These are the parts of Figure 26-12 that the plan currently leaves
implicit, partial, or simply absent. None are urgent; all are real.

### A — Slots have no declared order within a frame

The plan's `slots:` map has no `order:` or `sequence:` field. *SIZE
before SHAPE before COLOR* in Figure 26-12 is a property of the
frame; the plan's frames declare *which* slots exist and *who*
fills them, not *in what sequence*. This is the same gap noted as
gap C in [../ch1/1.2-sor.md](../ch1/1.2-sor.md), seen from the
frame side rather than the agency side.

### B — Ill-formed fillings are not detected as ordering errors

Because the plan has no slot order, it cannot reject a stimulus on
the grounds that the order of fills is wrong. Figure 26-12's *wooden
three heavy brown big first boxes* example is rejected by the
*frame*; the plan would reject only at `settle`, and only if the
result was incoherent enough to trigger `failure_conditions`.

### C — Slot groups (composite polynemes)

The figure brackets *big round white* under a single POLYNEME label.
The plan's frame schema has no `slot_group:` or `composite:` field.
A frame cannot say "these three slots are filled together by the
same agency, and together they form one symbol that other frames
can excite." Composite polynemes that span adjacent slots are not
representable.

### D — Polynemes against a *frame's internal structure*

Polynemes in the plan are matched against external surfaces — paths,
labels, phrases. Figure 26-12 uses *polyneme* for a structure *inside*
a frame. There is no polyneme kind whose `match:` field reads "these
slots of this frame, when all filled." Inter-frame excitation
mediated by groups of filled slots is not part of the schema.

### E — Type-based slot validation

Each labelled slot in Figure 26-12 expects a particular type
(*NUMBER* expects a numeral, *COLOR* expects a colour adjective).
Gap D from [26.2-sor.md](26.2-sor.md) recurs: the plan's slots
carry no type, so the frame cannot reject *three* in the *COLOR*
slot on type grounds.

### F — Ordering applies to *all* compositional units, not only frames

Figure 26-12 hints that the schedule idea extends to verb phrases,
clauses, and (by §26.9) vision. The plan applies order at the
pipeline phase scale and nowhere else: agencies, critics, censors,
and signals are *sets*, not *sequences*. The figure's broader claim
that *composition is scheduled* has only one implementation site.

## Implication for the plan (no changes proposed here)

Figure 26-12 is the figure that generalises *frame* into *schedule*:
a frame is not just a list of terminals, it is an ordered list of
terminals, some of which group into composite polynemes. The
implementation plan honours named slots and optional fills; it does
not yet honour order, slot groups, or composite polynemes that
span them.

The gaps above are recorded here as analysis, not as a change
request. Any move to close them — an `order:` field on slots, a
`slot_group:` or `composite:` field on the frame schema, or a
polyneme kind whose match condition is a filled slot group —
would touch the frame schema in [`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
the pipeline in [`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
and the protocol material in `THE-SOCIETY-OF-REPO/02-protocols/`,
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
