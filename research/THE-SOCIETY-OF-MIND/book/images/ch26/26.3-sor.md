# Figure 26-3 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [26-3.md](26-3.md) — *Two sentences filling the
present frame.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 26-3 and checks each
one against the implementation plan. The aim is to mark, plainly, what
the plan already absorbs and what it does not yet name.

---

## The ideas Figure 26-3 actually carries

Pulled from the figure and Minsky's surrounding text in §26.2
*understanding stories*:

1. **Slot-filling is incremental.** Sentence one fills *Host*;
   sentence two fills *Gift*. The frame survives a partial state.
2. **A single frame instance persists across inputs.** The right-hand
   box is the *same* frame as the left-hand box, not a new one.
3. **Re-reference resolves into the existing frame.** *He* in
   sentence two does not create a new *Host*; it lands on the *Host*
   already filled by sentence one.
4. **Surface form does not match slot order.** *Host* was filled
   first; the second sentence happens to fill *Gift* first and
   re-reference *Host* second. The frame imposes an order that the
   text does not.
5. **The constraint inherited from 26-2 keeps doing work.** The
   *Host likes Gift* relation from [26-2.md](26-2.md) is what carries
   the unstated inference *the kite is for Jack* across the two
   sentences.
6. **Comprehension is "merge" rather than "parse."** The figure
   captions describe the second sentence as *suggesting* fillers,
   not asserting them; comprehension is a sequence of proposed
   merges into a shared structure.
7. **Pronoun resolution is a frame operation.** Resolving *He* and
   *She* depends on knowing which slot they target, not on
   surface-string matching.

These seven items are what the rest of this file tests against the
implementation plan.

## What the implementation already absorbs

### 1 — Incremental, persistent state across a run

[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
and [`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
carry a workspace state object through the phase chain
(`perceive → activate → deliberate → criticize → censor → settle`).
That state is the operational analogue of the *same frame* persisting
across the two boxes of Figure 26-3.

### 2 — Slot-filling as a named operation

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
gives every frame a `slots:` map with `required:` and `filled_by:`.
Filling `relevant_files` then `proposed_patch` then `tests` is the
same shape as filling *Host* then *Gift*. The `code-change` frame
explicitly tolerates partial fills until `settle` rules on
completeness.

### 3 — Multiple stimuli into one frame

The signal schema in [`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md)
allows many signals to arrive into the same workspace before
settlement. Two issue comments, or an issue plus a PR review, can
both contribute to the same frame instance, matching Figure 26-3's
two-sentence merge.

## What the implementation does not yet take into account

These are the parts of Figure 26-3 that the plan currently leaves
implicit, partial, or simply absent. None are urgent; all are real.

### A — No first-class *frame instance* identity

The plan has frame *definitions* (`question.frame.yml`,
`code-change.frame.yml`, …) and a run-scoped workspace, but it has
no object that says "this is frame instance
`frame.code-change.<run-id>`, slot `relevant_files` filled, slot
`proposed_patch` empty." Without an instance id, you cannot ask "is
this the *same* present frame the previous comment was filling, or a
new one?" Figure 26-3's central claim — *the same frame across two
sentences* — has no schema-level representation.

### B — Re-reference / co-reference resolution

The figure's *He → Host* and *She → Mary* moves are pronoun
resolution against the current frame instance. Nothing in
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
or [`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
provides a co-reference step. References across signals — "the PR I
mentioned", "that workflow file", "the same bug" — would have to be
re-discovered from scratch each phase.

### C — *Suggest* vs *assert* in slot-filling

Figure 26-3's captions are careful: each sentence *suggests* a
filler. The plan's `slots:` field treats fills as commitments, not
suggestions. There is no notion of a *tentative fill* that another
agency can confirm or override before `settle`. The proposal layer
exists at the *settlement* level
([`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)),
not at the *slot* level.

### D — Constraint carried across the merge

Gap A from [26.2-sor.md](26.2-sor.md) applies twice as forcefully
here. The whole point of Figure 26-3 is that the inherited
constraint *Host likes Gift* does work *between* the two sentences,
producing the unstated conclusion *the kite is for Jack*. With no
`constraints:` field on a frame, there is nothing for that
between-stimulus inference to attach to in the plan.

### E — Continuation across runs

A frame can persist across two sentences in Minsky's example
because the *same reader* holds both. The plan's frame instance,
even if it existed (gap A), would be scoped to a single workflow
run. Comprehension that spans a thread of issue comments over
weeks — the realistic case for an issue tracker — needs an
inter-run frame-instance identity that
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
does not yet define.

## Implication for the plan (no changes proposed here)

Figure 26-3 is the figure that earns the word *understanding* by
showing comprehension as the merge of successive inputs into a
single, persistent, partially-filled frame. The plan honours
*partial fills* and *one workspace per run*, but does not yet
provide a frame-instance identity, a co-reference operation, or a
constraint-carrying field that would let inference travel between
inputs the way it travels between Figure 26-3's two sentences.

The gaps above are recorded here as analysis, not as a change
request. Any move to close them — frame-instance ids, a
co-reference resolution phase, tentative-fill semantics, or
cross-run frame continuity — would touch the frame schema in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
the pipeline in [`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
state and memory in [`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and the signal schema in [`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
