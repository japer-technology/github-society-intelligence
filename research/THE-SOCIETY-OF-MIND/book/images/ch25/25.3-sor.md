# Figure 25-3 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [25-3.md](25-3.md) — *Three views of one cube.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 25-3 and checks each
one against the implementation plan. The aim is to mark, plainly, what
the plan already absorbs and what it does not yet name.

---

## The ideas Figure 25-3 actually carries

1. **One object, many appearances.** The cube does not change; the
   image of the cube changes radically with viewpoint.
2. **Features come and go.** Face *A* is visible from the left and
   gone from the right; face *C* is the opposite. *Visible* is not
   *present*.
3. **Some features persist across views.** Face *E* shows up in all
   three. Persistence under transformation is the diagnostic for
   "same object."
4. **Naive perception would discard the past.** If each view started
   from scratch, every step of movement would be an identity crisis.
5. **A single percept must span the views.** Object-constancy is the
   work to be done; the figure exists to make the problem visible.
6. **The viewpoint is part of the state.** The three views are not
   three objects; they are one object indexed by viewpoint.
7. **The figure is a setup, not a solution.** It motivates frame-arrays
   (Figure 25-4) without itself solving anything.

## What the implementation already absorbs

### 1 — Stable identifiers across runs

[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
and `THE-SOCIETY-OF-REPO/02-protocols/01-identity.md` give every
agency, frame, K-line, and settlement a stable id. The same `agency`
is the same `agency` across runs even when its surface activations
differ — the "same object across views" pattern, at the agency
granularity.

### 2 — K-lines as memory of how things were assembled

K-lines in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
record which units worked together in a past run. That is one form
of "feature *E* recurs across views" — the K-line is what makes a
configuration recognisable when it appears again.

### 3 — Memory tier outlives a single run

`memory/` in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
is the persistence layer. By design, the next run can read what the
previous run wrote. The infrastructure for "do not start from
scratch" exists.

## What the implementation does not yet take into account

### A — No object-identity across runs

The plan persists *artefacts* (settlements, K-lines, frame instances)
but does not declare a kind for *objects under observation*. A user
asking about the same file three times produces three independent
runs; there is no `object-id` on the workspace that says "this is the
same thing we considered last time." Figure 25-3's *one cube* has no
schema slot.

### B — No registry of feature visibility per view

The figure makes "face *A* is visible from view 1, not from view 3"
explicit. The plan has no per-run record of which features of the
stimulus were attended to. `archivist` records what was *produced*,
not what was *seen and ignored*. So the next run cannot ask "did
face *A* show up last time?"

### C — Viewpoint is not a state field

There is no `state.viewpoint` or equivalent — no record of *which
slice of the object the current run is looking at*. The plan treats
each stimulus as self-contained. A run cannot say "this is the same
file from a different angle (e.g. via PR review vs. via direct
read)."

### D — No mechanism for cross-run feature accumulation

Figure 25-3's downstream claim, finished in Figure 25-4, is that
features seen in one view should remain available in another. The
plan has no "accumulating dossier" for an object across runs. Each
run starts from `memory/` reads it considers relevant, but there is
no schema-level *object dossier* that grows monotonically.

### E — Persistence-under-transformation is not a critic

Figure 25-3 implies a competence: "given these three views, decide
whether they are of the same object." No critic in the catalogue
([`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md))
does this. A `critic.same-object` that checks whether a new stimulus
is a new view of an already-tracked object does not exist.

### F — No viewpoint-aware retrieval

`kline-retriever` retrieves K-lines that match the current
configuration. It does not retrieve K-lines that match *the object
under consideration regardless of viewpoint*. The retrieval key is
the situation; in figure terms, that is a per-view key, not a
per-object key.

## Implication for the plan (no changes proposed here)

Figure 25-3 is a problem statement, not a solution. It asks the plan
to face a question it does not currently answer: *what is an "object"
that survives across runs, and how is it identified when its
appearance changes?* The plan persists agencies, K-lines, and
settlements with stable ids, but it does not represent the *subject*
of a run as an entity in its own right.

Any move to close these gaps — an `object-id` on the workspace, an
object-dossier kind under `memory/`, a `state.viewpoint`, a
`critic.same-object`, or a viewpoint-aware retrieval mode for
`kline-retriever` — would touch the state and memory machinery in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
the agency catalogues in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the frame and K-line machinery in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
and the identity scopes in
`THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`, and so falls
under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
