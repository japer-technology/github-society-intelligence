# Figure 25-2 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [25-2.md](25-2.md) — *Two interpretations of a single
cube.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 25-2 and checks each
one against the implementation plan. The aim is to mark, plainly, what
the plan already absorbs and what it does not yet name.

---

## The ideas Figure 25-2 actually carries

1. **Disambiguating cues do not abolish alternatives.** Adding shading
   to a Necker cube reduces, but does not eliminate, the second
   reading.
2. **The two readings are co-present as candidates, sequential as
   percepts.** Both readings are available at once; only one is held
   at once.
3. **A frame *claims* features.** Each face is assigned to a frame —
   *front* belongs to the upper view *or* to the lower view, never to
   both at the same time.
4. **The word "or" is doing real work.** Minsky's caption joins the
   two cubes with *or*, not *and*. Choice is the operation; co-presence
   is the prior.
5. **Cues are weights, not commitments.** Shading shifts the
   probability of which frame wins but does not turn the loser into
   nonsense.
6. **The chooser is implicit in the figure.** Nothing on the page
   shows *who* picks; the figure assumes a selecting agency.

## What the implementation already absorbs

### 1 — Frame catalogue offers alternatives

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
ships seven frames as siblings. A stimulus that could be read as a
`bug` *or* a `feature` is exactly the cube-with-shading case: two
licit readings, one of which must win.

### 2 — Selection is a named step

The transition from "candidates" to "bound frame" happens in
`perceive → activate`
([`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)).
The figure's `or` corresponds to this phase: the run chooses.

### 3 — Cues are routed through the perceivers

The perception family in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md)
is the set of units that turn raw stimulus into evidence for one
frame over another. That is the operational analogue of shading
making the upper-view cube the more likely percept.

## What the implementation does not yet take into account

### A — The losing reading is not retained

When `perceive` commits to a frame, the runner-up is not kept on the
workspace as a *candidate with a confidence score*. Figure 25-2's
whole point is that the other cube is still *there*; the plan deletes
it. See gap A in [25.1-sor.md](25.1-sor.md) — this is the same gap,
sharpened by the fact that 25-2 makes co-presence explicit.

### B — No record of which features each frame claimed

Even within the winning frame, the plan does not store "frame `bug`
claimed features X, Y, Z of the stimulus." The figure's per-face
assignment has no per-feature analogue in the manifest schema or in
the state schema in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md).
A later critic that wanted to ask "did the bug frame really claim
*this* sentence?" has nothing to read.

### C — No weighted scoring across frames

There is no schema in the plan for "frame `bug` scored 0.7, frame
`question` scored 0.3 against this stimulus." Selection is treated as
categorical. The figure's *cues as weights* model is not represented;
the perceiver does not output a distribution.

### D — Cue catalogue is implicit

The figure's "shading" is an explicit, named cue. The plan's
perceivers each apply their own heuristics; there is no shared
registry of *cues* that map stimulus features to frame weights — for
example, "`.forgejo/workflows/**` paths → +weight on
`security-sensitive`". The mapping exists in prose, not in data.

### E — No "near miss" memory

A run where `bug` barely beat `question` looks identical, in the
record, to a run where `bug` won unanimously. The figure says these
are different epistemic situations. The plan's `archivist` writes
the chosen frame, not the margin.

### F — The chooser is not named

Figure 25-2 assumes a selecting agency without showing it. The plan
*does* name perceivers, but it does not name a single
`agency.perception.frame-selector` whose job is the final `or`. The
choice is a phase outcome, not the output of a named unit.

## Implication for the plan (no changes proposed here)

Figure 25-2 differs from 25-1 by adding cues. Its operational message
is: *selection is weighted, not Boolean*. The implementation plan
selects, but it does not weigh visibly. Margins are lost, cues are
prose, runner-ups are discarded.

Any move to close these gaps — `state.frame_candidates` with scores,
a `cues/` registry under `agencies/perception/`, a named
`agency.perception.frame-selector`, or an `archivist` record of the
selection margin — would touch the frame machinery in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
the agency catalogues in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the state schema in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and the handoff schemas in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
