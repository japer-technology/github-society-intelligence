# Figure 24-4 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [24-4.md](24-4.md) — *A wall described by
compass-square zones.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 24-4 and checks each
one against the implementation plan. The aim is to mark, plainly,
what the plan already absorbs and what it does not yet name.

---

## The ideas Figure 24-4 actually carries

1. **A scene can be described by zone, not by pixel.** *Window at
   NW, shelves at NE, table at E* is enough; full detail is not.
2. **Compass labels are the activators.** *NW*, *N*, *NE*, …, *SE*
   carry the role; what is in the zone is whatever was noticed.
3. **A scene-frame holds nine slots, no more.** Nine zones is the
   right granularity for "what is on this wall"; finer detail goes
   somewhere else.
4. **Memory of a scene is memory of *what was where*, not memory of
   the scene.** Change detection reduces to comparing slot contents.
5. **Recognising rearrangement is cheap.** If the table moves from
   *E* to *W*, the comparison is one slot, not a whole image.
6. **The same nine-cell scheme transfers between domains.** The
   compass square is the direction-neme scheme of [24-3.md](24-3.md)
   relabelled, and prepares the picture-frame of [24-5.md](24-5.md).

## What the implementation already absorbs

### 1 — Description by zone-shaped labels

Path-polynemes in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
attach role labels (`workflow-file`, `policy-file`, `secret-file`)
to file paths in much the same spirit: the repository is described
by *which zone of meaning a file is in*, not by the file's
contents. The discipline of *labels carry the role, contents fill
the role* is honoured at the path layer.

### 2 — Comparing slot contents across runs

State and memory in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
keep prior settlements addressable per stimulus, and K-lines hold a
`useful_context.files_read`. Comparing *what filled which slot last
time* is operationally possible; what changed can be read off
durable artefacts.

## What the implementation does not yet take into account

### A — No scene-frame, no wall-frame

The first-ship frame catalogue covers situations
(`question`, `bug`, `feature`, `code-change`, `security-sensitive`,
`self-modification`, `novel`). None of them describe a *layout* or
a *state of the world* in the way the wall-frame does. The plan
has no frame whose job is to record *the present arrangement of
some space*.

### B — No spatial zone vocabulary

As noted in [24.3-sor.md](24.3-sor.md), the plan has no
direction-neme family. *NW*, *N*, *NE*, …, *SE* have no
counterpart in any polyneme catalogue and could not be referenced
by an agency today.

### C — Rearrangement is not a first-class concept

The plan detects *change* via diffs in git and via settlement
outcomes; it does not detect *rearrangement* in the figure's sense
(same parts, different zones). There is no critic, censor, or
agency in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md)
whose role is "compare zone contents across runs".

### D — Repository as a place is not modelled

The figure treats a wall as a place with zones. The plan treats the
repository as a graph of files with paths and labels. The two are
close in *operation* — both let you ask "what is at X?" — but the
plan has no place-with-zones model on which the wall-frame idea
could be projected.

### E — Coarse-grained memory of layout is absent

Memory in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
stores episodic, semantic, procedural, failure, and K-line entries.
None of these stores *the present coarse layout* of anything — for
example, *which top-level directories currently hold which kinds of
content*. The figure's main proposition (cheap layout memory) has
no current artefact.

## Implication for the plan (no changes proposed here)

Figure 24-4 makes the same argument as [24-3.md](24-3.md) but in a
more concrete domain: a small fixed vocabulary of zones, applied to
one scene, gives cheap and useful memory of *what was where*. The
implementation plan honours the part of this that already fits the
repo (zone-style path-polynemes) and does not yet carry the part
that would require a spatial or layout model.

Any move toward a scene-frame, a layout polyneme family, or a
rearrangement critic would change the frame and polyneme schemas in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
the memory layout in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and the agency catalogues in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
