# Figure 24-2 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [24-2.md](24-2.md) — *A complete Trans-frame as a
row of AND-agents.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 24-2 and checks each
one against the implementation plan. The aim is to mark, plainly,
what the plan already absorbs and what it does not yet name.

---

## The ideas Figure 24-2 actually carries

1. **A frame is a row of terminals under one frame-agent.** It is a
   composite, not a record; the frame-agent is itself a named unit
   that broadcasts to its terminals.
2. **Each pronome corresponds to exactly one terminal.** *Actor*,
   *Origin*, *Trajectory*, *Destination*, *Vehicle* each pick out
   one AND-gate in the row.
3. **Fillers are K-lines.** *John*, *Boston*, *Route 95*, *New
   York*, *Car* are not stored values; they are stored
   *activations* that the terminal can re-arouse.
4. **Asking the frame restores the filler.** Wondering about the
   Destination retrieves *New York* by re-firing the Destination
   terminal.
5. **Frames are cheap.** A Trans-frame is "nothing more than a row
   of AND-agents wired to K-lines"; the representational power comes
   from the wiring, not from heavy machinery per slot.
6. **The frame-agent is the activator of its own terminals.** There
   is no external controller; the unit that *is* the frame is the
   unit whose firing makes its terminals available.
7. **Slots have a fixed arity per frame kind.** The Trans-frame has
   exactly the five pronomes shown; one cannot ask it about
   *Colour* without changing the frame.

## What the implementation already absorbs

### 1 — Frames as named composites with named slots

The `code-change` frame in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
declares its required slots
(`user_goal`, `relevant_files`, `proposed_patch`, `tests`, `risks`,
`final_user_response`) and pairs each to `filled_by:` agent ids.
This is the same shape as Figure 24-2: one frame, several named
roles, each role pointing at the unit that can fill it.

### 2 — Restoring prior content by activation

K-line reactivation in the `activate` phase preloads
`useful_context.files_read` and boosts the agencies that previously
filled the frame. *Wondering about the Destination retrieves New
York* maps onto *restoring the K-line preloads what filled the
relevant slot last time*, at the granularity the plan supports.

### 3 — Cheap composition

The plan deliberately keeps the frame schema small (matches, slots,
defaults, failure conditions, linked K-lines). It does not put
heavyweight machinery per slot, which is consistent with the
figure's "very few primitives" claim.

## What the implementation does not yet take into account

### A — No frame-agent as a runtime unit

In Figure 24-2 the frame-agent is the horizontal bar at the top: it
is itself a unit that, when active, makes all its terminals
addressable. In the plan a frame is a YAML file; there is no agent
manifest behind it. Nothing in `.forgejo-society/agencies/` is "the
agent that *is* the code-change frame".

### B — Terminals are not named

The plan's slots are dictionary keys (`user_goal:` …). Figure 24-2
draws each terminal as its own unit. The plan has no per-slot
manifest, no per-slot id, and so no way for one agency to excite
"the Destination terminal of the code-change frame" as opposed to
exciting an agency that happens to write into the `relevant_files`
slot.

### C — Fillers are values, not K-lines

In the plan, a slot is filled by the output of an agency
(`filled_by: [<agent-id>, ...]`). The filler ends up in
`state/.../<slot>.json` (or the equivalent) as data. Figure 24-2's
filler is an aroused K-line — a reusable activation, not a value.
The plan does not currently store per-slot K-lines.

### D — No slot-level retrieval interface

Figure 24-2 supports queries of the form *what was the
Destination?*; the plan supports queries of the form *what was the
settlement?*. There is no agency or service in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md)
whose job is *re-fire the K-line attached to slot X of frame F*.

### E — Slot arity is enforced by schema, not by structure

The plan's frame schema lists slots; adding or removing a slot is a
schema edit. Figure 24-2's fixed arity comes from the picture
itself — the terminals are drawn. The plan does not gain any
structural guarantee that the *Destination* terminal cannot be
quietly reused for something else; this is enforced socially by
review, not by the wiring.

## Implication for the plan (no changes proposed here)

Figure 24-2 is the figure that earns the picture-frame architecture
that the rest of chapter 24 stacks on. The implementation plan
already shares the *outline* — named frames with named slots filled
by named agencies — but it represents slots as record fields rather
than as agents and fillers as values rather than as K-lines.

Any move toward the figure's shape — a `frame-agent` manifest kind,
per-slot terminal manifests, per-slot K-line storage, a *re-fire
slot X* operation — would change the frame and K-line primitives in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
the memory layout in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and the handoff schemas in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
