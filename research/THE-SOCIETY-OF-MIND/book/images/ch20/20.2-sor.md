# Figure 20-2 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [20-2.md](20-2.md) — *Selfridge's THE CAT.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 20-2 and checks each
one against the implementation plan. The aim is to mark, plainly, what
the plan already absorbs and what it does not yet name.

---

## The ideas Figure 20-2 actually carries

Pulled from the figure caption and Minsky's surrounding text in §20.3
*Visual ambiguity*:

1. **Identical low-level evidence, different high-level reading.** The
   same glyph is parsed as *H* in one position and *A* in another.
2. **Context decides what counts as the signal.** The choice is
   driven by the surrounding letters and the expected words.
3. **Top-down feedback into a lower agency.** A language-level
   expectation reaches back into a perception-level decision.
4. **The lower agency is unaware of the override.** Vision does not
   report "I saw an ambiguous glyph and language picked"; it simply
   reports *H* or *A*.
5. **Default expectations have priority.** *THE* and *CAT* are
   common; the system commits before any conscious deliberation.
6. **The same input can yield different commitments at different
   positions.** Position-within-context, not just the input itself,
   shapes the reading.
7. **Cross-agency influence is bidirectional in practice.** Perception
   is not a one-way pipe from sensor to interpretation.

These seven items are what the rest of this file tests against the
implementation plan.

## What the implementation already absorbs

### 1 — Context as an explicit activation source

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
treats frames as the standing context that biases which polynemes and
agencies activate. A frame already established at the time a new
input arrives shifts which perception agencies fire — this is the
mechanism that *could* host the Selfridge effect.

### 2 — Multiple agencies on one input

[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
explicitly fans `perceive` out across multiple perception agencies.
The plan does not assume a single perception agency owns a given
input.

### 3 — A `settle` phase that picks a reading

As in Figure 20-1, `settle` consolidates competing perception signals
into a single committed interpretation — the operational analogue of
the quiet *H*/*A* decision.

### 4 — Memory of past readings biases future ones

[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
keeps K-lines, and `agency.memory.kline-retriever` (named in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md))
can re-activate the same partial state on a similar input. That is
the right substrate for "the system has seen *THE* often, so prefer
*H*."

## What the implementation does not yet take into account

### A — No top-down loop from a downstream agency to perception

The pipeline in
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
runs `perceive → activate → deliberate → criticize → censor → settle`
in one direction. There is no specified mechanism by which a
`deliberate` or `criticize` step *re-runs perception with a hint*.
Figure 20-2's central move — language reaching down into vision — is
not representable inside a single run.

### B — Signals do not carry an ambiguity envelope

A perception signal asserts a reading. The plan does not specify a
`candidates: [...]` field listing the alternatives the agency
considered. Without that envelope, a downstream agency cannot say
"pick the second candidate instead" — it can only accept or reject.

### C — No critic that arbitrates by surrounding context

There is no `agency.critic.context-consistency` whose role is to
notice that two adjacent readings disagree with the standing frame
and request a re-reading of one. The closest plan element is
`critic.scope`, which guards run boundaries, not local
within-input coherence.

### D — Position-aware re-interpretation is absent

The plan does not model "the same glyph in two positions yielded two
readings." Signals are addressable by id, not by position-within-input
or position-within-context. Figure 20-2's *H*-at-pos-2 vs *A*-at-pos-2
distinction has no schema slot.

### E — Default-driven commitment is implicit

The plan does not name a "priors" object that says *THE* and *CAT*
are common. Frame weights and K-line activation strengths exist, but
there is no documented protocol for using them as soft defaults during
perception, only as biases on which frame is active.

### F — The bidirectional view is missing from the documents

[`02-workflow-design.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/02-workflow-design.md)
describes the workflow as a linear chain of phases. The
forward-only diagrammatic style of the plan is exactly the picture
Figure 20-2 is arguing against. The plan can host backpressure (B, C,
D would let it), but the architectural documents currently describe
the system as bottom-up.

## Implication for the plan (no changes proposed here)

Figure 20-2 is the figure that breaks the bottom-up reading of
perception. The implementation plan currently *describes* the
pipeline bottom-up, and the schema offered for signals and
settlements does not yet carry the alternative readings or position
data that a downstream hint would need.

The gaps above are recorded as analysis. Any move to close them — a
`candidates` envelope on perception signals, a `position_in_context`
field, an explicit `re-perceive` step, a priors object, or a
`critic.context-consistency` — would touch the manifest schema in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the pipeline in
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
the handoff schemas in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
the workflow design in
[`02-workflow-design.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/02-workflow-design.md),
and the identity scopes in `THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`,
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
