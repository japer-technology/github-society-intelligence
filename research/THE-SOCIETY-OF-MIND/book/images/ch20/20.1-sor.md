# Figure 20-1 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [20-1.md](20-1.md) — *nine blocks, three arches, or
one superarch.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 20-1 and checks each
one against the implementation plan. The aim is to mark, plainly, what
the plan already absorbs and what it does not yet name.

---

## The ideas Figure 20-1 actually carries

Pulled from the figure caption and Minsky's surrounding text in §20.3
*Visual ambiguity*:

1. **One arrangement, several legitimate parsings.** The same fixed
   evidence supports *block-level*, *arch-level*, and *superarch-level*
   readings simultaneously.
2. **Level of organisation is itself a choice.** Choosing a parsing is
   choosing a *level of description*, not just a label.
3. **The readings are not refinements of each other.** "Nine blocks"
   is not a coarser version of "one superarch"; they are independent
   carvings of the same input.
4. **Settling among readings is normally invisible.** The system picks
   one and the conscious observer never sees the alternatives.
5. **Conflict is the normal case, not the exception.** Ambiguity
   resolution is not a special mode; it runs on every input.
6. **The chosen parsing depends on what is already active above.**
   Which level wins is a function of standing context, not just of
   the input.
7. **A surfaced parsing is a commitment.** Once "three arches" is
   adopted, downstream work treats the scene as three arches and
   stops considering the other readings.

These seven items are what the rest of this file tests against the
implementation plan.

## What the implementation already absorbs

### 1 — Multiple coexisting interpretations at perception time

[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
and the perception agencies in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md)
explicitly allow several `agency.perception.*` units to emit signals
about the same input. Multiple parsings of a single trigger are
therefore not a special case but the ordinary output of the
`perceive` phase.

### 2 — A `settle` step that picks one reading

[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
gives `settle` the role of consolidating competing signals into a
settlement object that the rest of the run treats as authoritative.
This is the operational analogue of Figure 20-1's quiet selection
among block / arch / superarch.

### 3 — Levels of description as named scopes

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
treats *frames* as the named contexts that decide which polynemes are
in scope. A frame change is precisely a change of level of
description: the input has not moved, but the system now parses it
differently.

### 4 — A record of the chosen reading

[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
keeps the settled outcome, and `remember` writes it back as a K-line.
Once a parsing is committed, it survives and biases later runs — the
"invisible settling" of Figure 20-1 leaves an audit trail.

## What the implementation does not yet take into account

### A — Competing parsings are not preserved

The settlement object that exits `settle` records *the* chosen
interpretation. The implementation plan does not specify a field that
keeps the *runner-up* parsings alongside it. Figure 20-1's
block / arch / superarch are all simultaneously valid; in the current
schema only the winner is named, so a later run cannot ask "what was
the second-best reading of this input?"

### B — No explicit level-of-description type

`agency.perception.*` units emit signals, but no signal field declares
its *level of organisation* (atomic / grouped / aggregate). Two
perception agencies firing on the same trigger at different levels
look, in the signal stream, like two competing claims rather than two
compatible readings at different scales.

### C — No protocol for cross-level coherence

When `perception.layout-scanner` says "nine items" and
`perception.pattern-grouper` says "three groups of three", the plan
has no critic or censor whose job is to check that the two readings
are *consistent* — that the three groups, expanded, are the nine
items. `THE-SOCIETY-OF-REPO/02-protocols/13-hierarchy-and-summaries.md`
covers summarisation of outputs, not coherence of simultaneous
parsings.

### D — Frame selection is signal-driven, not parsing-driven

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
activates frames from signals, labels, and paths. There is no
mechanism by which an alternative *parsing* of the same input flips
the active frame. In Figure 20-1, deciding to see "one superarch"
*is* a frame change; in the plan, frame changes come from outside the
perception layer.

### E — Ambiguity is not a tracked property

A trigger that admits several equally good parsings is, in the plan,
indistinguishable from one with a single clear parsing — both
generate one settlement. There is no `signal.ambiguity_score` and no
critic that escalates when the settlement is contested. Figure 20-1's
core observation — that ambiguity is normal — is therefore present
operationally only as silence.

## Implication for the plan (no changes proposed here)

Figure 20-1 says that *every* perceptual input is ambiguous and that
the system normally hides the conflict. The implementation plan picks
*a* reading and commits to it — which is the right default — but
discards the alternatives and does not mark the input as having been
ambiguous in the first place.

The gaps above are recorded as analysis. Any move to close them — a
`level_of_description` field on perception signals, a
`runner_up_settlements` slot on the settlement schema, an
`agency.critic.cross-level-coherence`, or an `ambiguity_score` on
signals — would touch the manifest schema in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the runtime pipeline in
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
the handoff schemas in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
and the identity scopes in `THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`,
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
