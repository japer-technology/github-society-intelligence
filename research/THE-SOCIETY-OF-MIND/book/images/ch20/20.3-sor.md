# Figure 20-3 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [20-3.md](20-3.md) — *five different drawings of the
same description.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 20-3 and checks each
one against the implementation plan. The aim is to mark, plainly, what
the plan already absorbs and what it does not yet name.

---

## The ideas Figure 20-3 actually carries

Pulled from the figure caption and Minsky's surrounding text in §20.3
*Visual ambiguity*:

1. **A description groups what surface features separate.** Five
   drawings, parametrically different, collapse to one structural
   description.
2. **Same-ness is a property of the description, not the data.** The
   drawings are "the same" only relative to a chosen description.
3. **Multiple descriptions of one input are coexistent.** The
   parametric description and the structural description are both
   true at once.
4. **Choosing the description chooses the equivalence class.** The
   set of "matching inputs" shifts with the description in use.
5. **Generalisation rides on descriptive abstraction.** Recognising
   the fifth drawing as the same kind is a generalisation step driven
   by the description.
6. **The complement of Figure 20-2.** There: one drawing read two
   ways. Here: five drawings read as one. Both depend on what
   description is active.
7. **The active description is itself a piece of state.** Whether
   "triangle with two extensions" is in use determines what gets
   matched.

These seven items are what the rest of this file tests against the
implementation plan.

## What the implementation already absorbs

### 1 — Descriptions are first-class as frames and signals

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
makes frames the named, structured descriptions of a situation. A
frame is exactly the kind of object that decides which inputs count
as "the same".

### 2 — Perception agencies that match by pattern, not by surface

The first-ship catalogue in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md)
includes pattern-shaped agencies — `patch-imaginer`, `diff-skeptic`,
`kline-retriever` — that work on structural features of diffs and
paths, not on raw bytes. They already implement the spirit of
"described, not pixel-matched."

### 3 — K-lines as reusable descriptions

[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
treats K-lines as the persistent descriptions of past situations,
which is the right home for "the description that made these five
drawings look alike."

### 4 — Multiple agencies on one input

As in Figures 20-1 and 20-2, `perceive` is fan-out, so multiple
descriptions of one input coexist by construction.

## What the implementation does not yet take into account

### A — No notion of an *equivalence class* over inputs

The plan can say "these signals were emitted on this input." It does
not specify a structure that says "these five distinct inputs are
instances of the same description, and therefore equivalent for
routing purposes." Figure 20-3's collapse of five drawings into one
class has no schema slot.

### B — No "active description" object

A frame is active, and signals are active. There is no documented
object that names *the description currently used by this perception
agency* — the moveable lens through which sameness is decided. The
description is implicit in the agency's prompt body rather than
addressable.

### C — Generalisation is not a named pipeline phase

The phase chain in
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
includes `perceive`, `activate`, `deliberate`, `criticize`, `censor`,
`settle`, `act`, `remember`, `report`. None of these names a
*generalise* step — "infer the description that would let this input
and the recent K-lines be treated as one." The work can happen inside
`deliberate`, but it is not separated out.

### D — No critic that polices descriptive cost

A description can be made arbitrarily inclusive ("anything with a
line in it"). There is no documented critic whose role is to reject
descriptions that are too coarse or too narrow for the run's
purpose. The plan trusts the prompt body of each agency to set the
right level.

### E — Equivalence is not exposed to `act` or `remember`

If `kline-retriever` retrieves a K-line because the current input
matches the description of an old one, the settlement records "this
K-line was used" but not "these two inputs were judged equivalent
under description D." The trail is *which K-line*, not *which
equivalence*.

### F — Cross-agency description sharing has no protocol

When `perception.layout-scanner` and `perception.pattern-grouper`
agree that two inputs are "the same kind of arrangement," there is no
documented protocol for them to converge on a shared description.
Each agency forms its own.

## Implication for the plan (no changes proposed here)

Figure 20-3 says that *what counts as the same input* is decided by
the active description. The implementation plan honours descriptions
as frames and K-lines, but does not yet expose either *the
equivalence class* or *the description in use* as addressable
objects, and has no phase that generalises across recent inputs.

The gaps above are recorded as analysis. Any move to close them — an
`equivalence_class` field on K-lines, an `active_description`
property on agency state, a `generalise` step in the pipeline, or an
`agency.critic.description-fit` — would touch the manifest schema in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the pipeline in
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
the frames protocol in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
the memory model in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and the identity scopes in `THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`,
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
