# Figure 16-7 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [16-7.md](16-7.md) — *Separate memory banks per
goal.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas Figure 16-7 carries and checks each
one against the implementation plan.

---

## The ideas Figure 16-7 actually carries

1. **One private memory per goal.** Each proto-specialist has its
   own store.
2. **Write-gating by active specialist.** Only the agent currently in
   control may write into its own bank.
3. **Read-isolation as well as write-isolation.** Hunger does not
   look into Thirst's memory at all.
4. **Memories never contaminate.** Lessons learned under one drive
   cannot drag another drive off-course.
5. **The cost is duplication.** Many banks, each holding overlapping
   facts about the world.
6. **The gate is part of the agency, not the memory.** Each
   specialist owns its access control.
7. **No transfer learning.** What Hunger learns helps only Hunger;
   nothing flows sideways.
8. **Stable identity through stable memory ownership.** A
   specialist's memory is *what makes it itself* over time.

## What the implementation already absorbs

### Memory is partitioned

[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
divides the memory tree by *class* (`klines/`, `frames/`,
`global-workspace/`, settlements, etc.) and by *stimulus*. There is
real structural isolation, even if not along the figure's axis.

### Per-stimulus state directories

Per-stimulus state under `.forgejo-society/state/<stimulus_id>/`
([`11-mapping-sor-to-implementation.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/11-mapping-sor-to-implementation.md))
guarantees that what one run writes during deliberation does not
silently leak into another run's scratch. This is a partial
analogue to "one bank per active episode."

### Write-gating exists at the authority layer

Every agency manifest carries an `authority:` field with values
`read | draft | propose | act | govern | human`
([`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md)).
The "only the active specialist writes" idea exists in the form
"only an agency with `act` authority writes."

---

## What the implementation does not yet take into account

### A — No per-agency private store

There is no `memory/<agency>/` directory. Every agency reads from
and writes to (subject to authority) the *same* memory tree. An
agency cannot have a private K-line file that no other agency may
read. The plan's partition is by *class* and *stimulus*, never by
*owner*.

### B — No read-isolation

The plan's posture is that memory is auditable, which implies
broadly readable. There is no field like:

```yaml
memory_visibility:
  read_by: [<agency-id>, ...]
  write_by: [<agency-id>, ...]
```

So Figure 16-7's "Hunger cannot even see Thirst's memory" is not
representable.

### C — No drive-keyed memory

Because drives are not first-class (see [16.2-sor.md](16.2-sor.md)
gap A), there is no `memory/drives/<drive>/` partition either.
"What did this drive learn?" cannot be asked of the store directly.

### D — Contamination is left to critics

The plan's defence against memory contamination is *review by
critic and censor*, not *structural prevention*. `critic.evidence`
and `censor.*` examine what is being written. The figure prevents
contamination at the wiring level; the plan detects it after the
fact.

### E — Active-specialist gating is absent

Figure 16-7's gate fires only when its owning specialist is active.
The plan's authority is a *static* property of an agency: a `draft`
agent may always draft, regardless of whether it "owns the moment."
There is no runtime concept of "currently in control" that gates
writes.

### F — Trade-off of duplication vs. sharing is not stated

The figure exists to motivate [16-8.md](16-8.md). The plan jumps
straight to a shared store and does not document the duplicated-
banks alternative it has implicitly ruled out, nor what would be
lost by ruling it back in.

## Implication for the plan (no changes proposed here)

Figure 16-7 names a different memory architecture from the one the
plan adopts: isolation by owner rather than by class. The plan has
deliberately chosen the shared design of [16-8.md](16-8.md) and pays
its cost with critics, censors, and authority levels. Adding
per-agency or per-drive private stores, read-isolation,
active-specialist write gating, or memory ownership as part of
identity would touch the memory layout in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
the manifest schema in [`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
and the identity protocol in
`THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`, and so falls
under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
