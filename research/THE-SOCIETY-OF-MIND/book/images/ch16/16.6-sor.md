# Figure 16-6 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [16-6.md](16-6.md) — *Cross-exclusion among
competing agents.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas Figure 16-6 carries and checks each
one against the implementation plan.

---

## The ideas Figure 16-6 actually carries

1. **Mutual inhibition as the arbitration primitive.** Every agent
   inhibits every other in a dense mesh.
2. **Noncompromise.** The point is *one winner*, not a weighted
   average. The mechanism is designed to suppress compromise.
3. **Avalanche dynamics.** A small lead becomes a total win, because
   each agent's outward inhibition weakens its rivals' inhibitions
   of it.
4. **Latched short-term memory.** The winning agent stays the winner
   until a strong external signal flips the group.
5. **Local rules, global outcome.** No central arbiter is drawn; the
   winner emerges from pairwise wiring.
6. **Symmetry by default.** The mesh is uniform — every agent has
   the same kind of connection to every other.
7. **External inputs can break the latch.** The system is not closed;
   a strong enough stimulus reshuffles which agent leads.
8. **Inhibition is bidirectional but separable.** A → B and B → A are
   distinct connections in the mesh.

## What the implementation already absorbs

### Pairwise inhibition exists in the schema

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
defines polynemes with both `excite:` and `inhibit:` maps:

```yaml
excite:
  <agent-id>: <weight 0..1>
inhibit:
  <agent-id>: <weight 0..1>
```

The wiring primitive Figure 16-6 needs — "agent A reduces the
likelihood of agent B firing" — is present.

### Symmetry is allowed

Nothing in the schema prevents an agency from inhibiting its peers.
The structure permits the dense mesh of Figure 16-6 in principle,
even if no first-ship catalogue draws one.

### Noncompromise at the settlement scale

[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
and [`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md)
describe `settle` choosing *one* settlement per stimulus, with a
single `reality_revision` recorded on merge. At the macroscopic
scale, the plan picks a winner rather than averaging.

## What the implementation does not yet take into account

### A — No avalanche dynamics

`excite:` and `inhibit:` weights are documented as static values
applied at activation. There is no iterative loop that lets a small
lead in activation strength feed back and amplify itself, and no
fixed-point procedure that converges on a single winner. The
activation model is one-shot, not relaxation-to-equilibrium.

### B — No latched state across runs

Figure 16-6's "winner stays the winner" is a *memory* property of
the inhibition network itself. The plan keeps no such latch.
Activation state is recomputed from scratch for every stimulus,
because [`02-workflow-design.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/02-workflow-design.md)
keys concurrency on the stimulus. There is no
`memory/cross-exclusion/<group>.yml` recording "Hunger held the
mesh as of the last run."

### C — No declared exclusion groups

The figure draws a *group* of mutually inhibiting agents. The plan
has no `exclusion-groups.yml` file, no `group:` field on agencies,
and no schema for "these N agencies form a winner-take-all set."
Inhibition is currently described only as pairwise weights between
named agents.

### D — Cross-exclusion is not used for arbitration

Even though `inhibit:` exists, no first-ship catalogue uses it to
arbitrate among competing proposals. Within a run, multiple
imagination branches and multiple drafts coexist; choice is made by
`settle` examining their content, not by competitors having
suppressed each other beforehand.

### E — No "strong external signal flips the latch" rule

Because there is no latch (gap B), there is also no rule for when an
external stimulus is *strong enough* to override the prior winner.
Such a rule would require both a persistent activation state and a
threshold field on polynemes — neither exists.

### F — No safeguard against full deadlock

A dense symmetric mesh can settle into a stalemate if every agent
inhibits every other equally. The figure assumes biological noise
breaks ties. The plan has no equivalent (no random tiebreak, no
fallback to `agency.meta-admin.*`), because it has no relaxation
loop in which a deadlock could occur.

## Implication for the plan (no changes proposed here)

Figure 16-6 is the canonical Society-of-Mind arbitration mechanism.
The implementation plan has the *parts* (`excite:`, `inhibit:`,
`settle`) but does not yet wire them into the figure's dynamic:
iterative competition, winner latching, declared exclusion groups,
strength-based override. Adding any of these would touch the
activation protocol in [`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
the runtime pipeline in [`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
the memory layout in [`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and likely the protocols under
`THE-SOCIETY-OF-REPO/02-protocols/`, and so falls under the "stop
and ask" rules in [`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
