# Figure 16-3 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [16-3.md](16-3.md) — *Inside the Thirst
proto-specialist.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas Figure 16-3 carries and checks each
one against the implementation plan.

---

## The ideas Figure 16-3 actually carries

1. **One need contains a whole mini-mind.** Inside *Thirst* there is
   a sensor, a recognizer, a planner, and effectors — a full
   perceive-decide-act loop.
2. **Detection is internal.** The first part is a *dryness sensor*:
   the agent watches its own state, not the world.
3. **Recognition is separate from detection.** Knowing one is thirsty
   is not knowing what a cup is. The figure splits these.
4. **Planning is a distinct stage.** A *planner* selects an approach
   between recognition and action.
5. **Action is the last stage.** Effectors arrive at the bottom; the
   chain is one-directional toward output.
6. **Per-need replication is expensive.** Read together with
   [16-2.md](16-2.md), the figure exists to make the cost of
   "everyone has their own everything" visible.
7. **The internal layering matches the outer chapter pattern.** Even
   one specialist contains, in miniature, the same sense-recognise-
   plan-act stack the chapter applies at the whole-animal level.

## What the implementation already absorbs

### A perceive-decide-act loop at the workflow scale

The phase chain in [`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md)
— `perceive → activate → deliberate → criticize → censor → settle →
act` — is exactly the sense → recognise → plan → act layering of
Figure 16-3, but lifted to the run level rather than embedded in one
agency.

### Recognition is its own family

`agencies/perception/` (and the `kline-retriever` and frame
selection in [`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md))
separates the "what is this stimulus?" step from the "what should we
do?" step, matching the figure's separation of recognizer from
planner.

### Action is gated and last

[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md)
and the `act` phase make effector use the final step, after
deliberation and censorship. The figure's downward arrow toward
action is honoured at the workflow level.

## What the implementation does not yet take into account

### A — No internal stages inside an agency

Figure 16-3 puts sensor / recognizer / planner / effector *inside
one agent*. In the plan, each of those roles is a separate agency
manifest at the same level. There is no schema field like:

```yaml
stages:
  - sense
  - recognise
  - plan
  - act
```

An agency is treated as atomic from the outside, with a single
prompt body. The recursive "one specialist is itself a mini-mind"
shape is not representable.

### B — No internal-state sensor

The first box in the figure watches *internal dryness*. The plan's
perception agencies watch *external* stimuli (paths, labels,
phrases, events). There is no agency family whose job is to read the
society's own state — for example, "the memory tree is unusually
large," "settlements are piling up unaccepted," "K-line retrieval
rates have dropped." The closest analogue is the meta-admin family,
but its members react to *requests* to self-modify, not to
introspected drift.

### C — Planner as a named role

The figure makes *planner* a distinct member of the agency. The
implementation absorbs planning into agency prompts (e.g.
`patch-imaginer` plans implicitly while drafting). There is no
`agency.<family>.planner` per family, and no schema field
distinguishing a planning step from an imagining step.

### D — Action attached to a need

In the figure, effectors are owned by *Thirst*. In the plan, action
is a workflow phase and a per-agency `authority:` line; it is not
tied to a long-running motivational owner. There is no record of
"this commit was an act of Safety" the way the figure suggests every
action is an act of one specific need.

### E — Per-agency recognition models

Figure 16-3 implies *Thirst* contains its own model of what counts
as water and what counts as a cup. The plan ships *shared*
recognisers (path-polynemes, label-polynemes, frames) and reuses
them across agencies. The per-agency private recogniser idiom — one
agency knowing things no one else knows — is not provided by the
current memory and frame layout, and would clash with the
shared-store design of [`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md).

## Implication for the plan (no changes proposed here)

Figure 16-3 is a fractal claim: the same sense-recognise-plan-act
shape that organises the whole animal also organises each one of its
needs. The implementation plan honours that shape *once*, at the
workflow level, and not at the agency level. Adding nested stages
inside an agency, an internal-state sensor family, a distinct
planner role, or per-agency private recognisers would touch the
manifest schema in [`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the activation protocol in [`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
and the memory layout in [`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
