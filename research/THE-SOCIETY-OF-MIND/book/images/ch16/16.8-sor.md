# Figure 16-8 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [16-8.md](16-8.md) — *Specialists sharing a general
memory.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas Figure 16-8 carries and checks each
one against the implementation plan.

---

## The ideas Figure 16-8 actually carries

1. **A single common store.** All proto-specialists read from and
   write to one pool.
2. **Economy.** One store is far cheaper than N private banks.
3. **Cross-specialist benefit.** What one agent learns is available
   to all the others without explicit transfer.
4. **A new hazard: structural damage.** Any specialist may rearrange
   the store in ways another relies on.
5. **Small agents cannot understand the whole store.** No specialist
   knows enough about the others to write *safely* on their behalf.
6. **Exploitation as the only available strategy.** Because writing
   carefully is impossible, specialists must instead *call* each
   other (the move that 16-9 makes explicit).
7. **The store is the substrate of cooperation, not the cooperator
   itself.** It is a passive arena; the agents do the work.
8. **One arena, many tenants.** Memory becomes shared infrastructure
   with the political problems that implies.

## What the implementation already absorbs

### One memory tree, many readers/writers

[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
and [`11-mapping-sor-to-implementation.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/11-mapping-sor-to-implementation.md)
ship a single `.forgejo-society/memory/` tree (`klines/`, `frames/`,
`global-workspace/`, `decisions/`, …) shared by every agency. This
is exactly Figure 16-8's pool.

### Cross-specialist benefit via K-lines

K-lines under `.forgejo-society/memory/klines/<class>/<date>-<stimulus>.yml`
are designed to be reusable across runs and across agencies. The
`agency.code.kline-retriever` exists specifically so that what one
run remembers, another run can recall, regardless of which agency
originally wrote it.

### Structural-damage hazard is acknowledged

`critic.*` and `censor.*` exist partly because writes to the shared
tree may break invariants others rely on
([`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md)).
The hazard in idea 4 is named; defences exist.

### Branch-as-imagination prevents direct damage to `main`

The branch protocol `society/<stimulus_id>/candidate-<n>`
([`02-workflow-design.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/02-workflow-design.md),
[`11-mapping-sor-to-implementation.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/11-mapping-sor-to-implementation.md))
means most writes never touch the shared trunk until they have been
critiqued and settled. The "any specialist may rearrange the store"
risk is mitigated by branching, even though it is not eliminated.

### Authority levels for write access

`read | draft | propose | act | govern | human` distinguish who may
actually modify the shared store, providing one axis of safety
Figure 16-8 leaves unsolved.

## What the implementation does not yet take into account

### A — No shared-memory invariants

Figure 16-8's worry is that a specialist may break a structure
*another* relies on. The plan has no machine-readable invariants
attached to memory regions — no schema saying "this directory must
remain index-sorted," "K-line ids in this class must follow this
pattern," "this frame slot must always be filled." Critics enforce
correctness by reading; nothing enforces it by construction.

### B — No write-by-write audit linking agency to mutation

[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
relies on git for history, but there is no explicit
`memory/<path>.write-log.yml` saying "agency X mutated this region
on date Y for stimulus Z." The "many tenants in one arena" reality
of Figure 16-8 is governed implicitly by commit metadata, not by an
explicit per-region register.

### C — No structural exploitation pattern

Figure 16-8 ends by saying the *only* available collaboration
strategy is exploitation — calling other agents. The plan does
provide direct addressing (`society <target>:` via
`directive.spock` and friends in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md))
but does not frame call-the-other-agent as the *primary substitute*
for write-on-behalf-of-others. The current substitute is more often
"submit a proposal that touches the file."

### D — No quotas on the shared store

In Figure 16-8 the store is a finite arena. The plan does not
budget per-agency or per-run writes to memory the way it budgets
tool calls and wall clock. A single misbehaving agency could in
principle fill the K-line store; the only check is critic review,
not a structural quota.

### E — Memory as substrate vs. memory as participant

Figure 16-8 is careful that the store is *passive*. The plan's
`global-workspace/<stimulus_id>.md` is borderline active — the
`conscious-presenter` reads it, summary tiers shape it. The
distinction "store is arena, not agent" is not stated as a design
constraint; nothing in the documentation prevents memory from
quietly accruing logic.

### F — No mention of the abandoned alternative

As in [16.7-sor.md](16.7-sor.md), the plan adopts the shared design
without documenting the per-agency-store alternative that was set
aside. A reader of the implementation plan cannot easily see *why*
the trade was made.

## Implication for the plan (no changes proposed here)

The implementation plan is already in Figure 16-8's regime: one
arena, many tenants, defended by critics, censors, and imagination
branches. What the figure highlights as still owed are *structural*
protections — machine-readable invariants, per-region audit logs,
explicit memory quotas, an explicit "store is passive" rule, and a
documentation of the trade against the per-agency design. Adding any
of these would touch the memory layout in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
the handoff schemas in [`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
and the safety policies in [`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
