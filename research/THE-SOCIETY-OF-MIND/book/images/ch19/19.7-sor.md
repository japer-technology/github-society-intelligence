# Figure 19-7 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [19-7.md](19-7.md) — *Connected versus disconnected
patterns.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This is the *Perceptrons* counterexample. It is a figure about a
limit, not a mechanism, so this analysis is mostly about whether the
plan inherits the limit or escapes it.

---

## The ideas Figure 19-7 actually carries

1. **Feature-bag representations lose structure.** Two patterns with
   identical heaps of fragments can still be qualitatively different.
2. **Some predicates are inherently relational.** *Connected* is not
   a property of a pixel; it is a property of how pixels are linked.
3. **No weighting will fix this.** The limit is not in the cut-off
   number but in the input representation.
4. **You must represent the relations themselves.** A recogniser that
   can see *connected* needs to see edges-between-fragments as
   inputs, not just fragments.
5. **Recognition has a representational floor.** Even an unlimited
   bank of weighted recognisers won't compute what its inputs do not
   encode.
6. **The fix is structural, not numeric.** It pushes the architecture
   toward frames, graphs, and named relations.
7. **This figure is a warning, not a tool.** Nothing in it is to be
   built; it is to be remembered when building.

---

## What the implementation already absorbs

### 1 — Frames carry structure that bags do not

Frames in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
have *named slots* (`user_goal`, `relevant_files`, `proposed_patch`,
`tests`, `risks`, …). The structure of a stimulus is not collapsed
into an unordered feature bag; it is unpacked into slots with names
and required-flags. That is the right *direction* for avoiding the
Perceptrons trap.

### 2 — Handoffs carry typed, structured payloads

Signal and handoff schemas in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md)
preserve relations between fields rather than flattening them. An
agency that consumes a handoff sees a structured object, not a
keyword bag.

### 3 — Settlements are structured artifacts

Settlement records carry typed sections rather than free text. The
governance layer in
[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md)
treats settlements as inspectable structure, not as a heap of
features.

## What the implementation does not yet take into account

### A — No explicit relational matchers

Polyneme `match` blocks list paths, labels, and phrases. There is no
`relation:` field — no way to say "match when path A *and* path B
both appear *and* B imports A." The figure's *connected* predicate
has no schema home.

### B — Graph-shaped state is absent from manifests

State in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
is files and K-lines, not edges. There is no first-class repository
graph (file → import → file, agency → activates → agency) that
recognisers could read.

### C — The pipeline does not compute relations explicitly

`perceive` returns features; `deliberate` consumes them; nowhere is
there a phase that says "compute the relations among these features
before judging them." Whatever relational competence exists comes
from LLM agents reading the structured handoff, which is exactly the
implicit, non-architectural fix the Perceptrons critique warns
against.

### D — No documented awareness of the limit

There is no "Perceptron warning" in the implementation prose: no
note that says "if you ever replace this LLM step with a weighted
matcher, the following relational predicates will silently fail."
The figure's warning is not recorded in the design.

### E — Hierarchical summaries do not recover relations

`agencies/assembly/*` and summary tiers fold many small signals into
one larger brief, but the folding step itself is not a relational
recogniser; it is a *summariser*. Going from many fragments to one
report is not the same as detecting *connected*.

## Implication for the plan (no changes proposed here)

Figure 19-7's force is preventive. The plan is, by accident or
design, *mostly* on the right side of its warning — frames and
handoffs preserve structure — but the warning itself is not written
down, and there is no schema for declaring relational matchers. If
any future move replaces an LLM step with a cheaper matcher, the
absence of (A), (B), and (D) above will matter.

Recording the warning would be a small documentation change inside
[`07-policies-and-safety.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/07-policies-and-safety.md)
or
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md);
adding relation matchers or graph-shaped state would touch the
matcher and state schemas in those files and in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
