# Figure 13-1 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [13-1.md](13-1.md) — *Four kinds of arch.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 13-1 and checks
each one against the implementation plan. The aim is to mark, plainly,
what the plan already absorbs and what it does not yet name.

---

## The ideas Figure 13-1 actually carries

Pulled from the figure caption and Minsky's surrounding text in §13.1
*Reformulation*:

1. **A category whose members share no measurable feature.** SINGLE,
   TOWER, LEAN, and WEDGE all deserve the name *arch* yet have no
   common edge, block-count, or silhouette.
2. **Uniframe failure as a diagnostic.** When no single description
   in the available vocabulary covers all members, the failure is
   not the learner's — it is the vocabulary's.
3. **Vocabulary as a choice variable.** The set of admissible
   descriptions ("blocks", "edges", "stacks") is not given; it is
   itself something the mind can replace.
4. **The category name precedes the schema.** *Arch* is already in
   use as a label before any uniframe for it exists. The label is
   doing classification work that the description cannot justify.
5. **Heterogeneity is the rule, not the exception.** A real-world
   concept will normally collect instances that look nothing like
   one another at the block-by-block level.
6. **Reformulation is required, not optional.** The figure is set
   up so that no amount of fiddling with the original vocabulary
   will close the gap. Only changing what counts as a part will.
7. **A row of examples is itself a representation choice.** Placing
   four arches side by side is already a uniframe move — the figure
   asserts they belong together before showing why.

## What the implementation already absorbs

### 1 — Labels and ids are first-class, independent of schema

[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md)
gives every unit a stable id (`agency.code.patch-imaginer`,
`critic.evidence`, `censor.cloud-egress`) before saying anything
about what its body must look like. The plan can name a category
without yet having a uniframe for it — Figure 13-1's point 4.

### 2 — Manifests are deliberately small and pointed

The first-ship catalogues — `patch-imaginer`, `diff-skeptic`,
`kline-retriever` — describe specialists, not generalists. They are
the implementation's acknowledgement that one description rarely
covers a wide concept; better to have several narrow ones.

### 3 — Frames as a vocabulary slot

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
treats frames as the unit of "current vocabulary." A frame
(`frame.self-modification`, `frame.regression-debug`) brings a set of
slots and expected fillers, which is the operational form of "the
vocabulary in which a uniframe could be written."

## What the implementation does not yet take into account

### A — No mechanism for uniframe failure

Nothing in the runtime tracks "we keep trying to describe X with
vocabulary V and the description does not fit." Failure to classify
shows up only as `critic.scope` complaints or `settle` refusals; it
is not aggregated into "the vocabulary needs replacing." There is no
schema field like `frame.unsatisfied_count` and no critic whose job
is to flag chronic miss-fits.

### B — No notion of *reformulation* as a runtime move

The plan has `frame.self-modification`, but that frame is about
changing *manifests*, not about changing the *vocabulary in which
manifests are written*. Replacing "blocks" with "body + supports"
— Minsky's move in this chapter — has no operational analogue.
There is no `agency.meta-admin.reformulator` and no event kind
`event.vocabulary.replaced.*`.

### C — Heterogeneous category membership is not expressible

A manifest's `activates_on:` is a list of literal triggers (labels,
paths, phrases). It cannot say "this agency answers to *arch*-shaped
situations whose instances need not share any trigger." The plan
assumes categories are closed under a feature set; Figure 13-1 is
about categories that are not.

### D — Vocabulary is implicit, not addressable

Frames carry slots, but the *set of admissible slot names* across
the system is not itself a named, versioned object. There is no
`vocabulary.v1` → `vocabulary.v2` distinction in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md);
schema evolution happens through manifest edits and git history,
not through a first-class reformulation record.

### E — A category label without a uniframe has nowhere to live

If a maintainer wanted to record "we have started using the word
*arch* but have no shared description yet," there is no folder for
it. `agencies/`, `critics/`, and `censors/` all require a manifest
body. A pre-schema concept would have to be staged in
[`07-workspace/active-settlements/`](../../../../../THE-SOCIETY-OF-REPO/07-workspace/active-settlements/)
indefinitely.

## Implication for the plan (no changes proposed here)

Figure 13-1 is the figure that earns the chapter by exhibiting a
category the existing vocabulary cannot save. The implementation
plan honours specialisation and stable naming, but offers no
machinery for noticing that a vocabulary has run out and no place
to record a category that exists before its schema does.

The gaps above are recorded here as analysis, not as a change
request. Any move to close them — for example, a vocabulary-version
record in [`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
a `agency.meta-admin.reformulator`, or a critic that aggregates
chronic miss-fits — would touch
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
and the identity scopes in `THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`,
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
