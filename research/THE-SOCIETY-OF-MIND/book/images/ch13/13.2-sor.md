# Figure 13-2 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [13-2.md](13-2.md) — *Body and supports drawn over
the four arches.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 13-2 and checks
each one against the implementation plan. The aim is to mark, plainly,
what the plan already absorbs and what it does not yet name.

---

## The ideas Figure 13-2 actually carries

Pulled from the figure caption and Minsky's surrounding text in §13.1
*Reformulation*:

1. **A new vocabulary as a single template.** *Body + supports* is a
   tiny schema that replaces the previous catalogue of block-level
   descriptions.
2. **The schema cuts across what the world supplies.** The dashed
   line need not coincide with any real edge — its work is to *impose*
   a boundary the world did not provide.
3. **The same template applies to every member of the category.**
   Once the line is drawn, SINGLE, TOWER, LEAN, and WEDGE all admit
   the same three-region reading.
4. **Reformulation succeeds by adding parts, not by changing the
   instances.** The arches themselves are unchanged; what changes is
   the description hovering over them.
5. **The lower-level description is faded, not deleted.** The old
   block edges still exist; they have been demoted from foreground
   to background while the new template runs.
6. **A template fixes both partition and role.** *Body* and
   *support* are not merely regions; they are typed slots with
   expectations (one bears, the others hold).
7. **The reformulation is reversible.** Erasing the dashed line
   restores the previous reading without losing information.
8. **Reformulation is uniformly applied, not negotiated case by
   case.** The same overlay is drawn over all four arches in one
   stroke; there is no per-instance argument about where the body
   should be.

## What the implementation already absorbs

### 1 — Frames provide typed slots

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
defines a frame as a small bundle of named slots with expected
fillers. *Body + supports* is structurally the same shape — a
three-slot frame applied to whatever the perception layer produces.
Point 6 (template fixes partition and role) is honoured.

### 2 — Summary tiers as new layers over old detail

`agencies/assembly/*` and the summary-tier protocol described in
`THE-SOCIETY-OF-REPO/02-protocols/13-hierarchy-and-summaries.md`
collapse detailed signals into briefs. This is point 5
(lower-level remains, but is demoted) applied to *outputs*. The
detail is preserved in git; the brief sits above it.

### 3 — Activation as a uniform overlay

A frame's `excites:` and the polyneme machinery in
[`06-*`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
let the same trigger fire the same agencies across heterogeneous
situations. This is the operational shadow of point 8 (the same
overlay applied uniformly).

## What the implementation does not yet take into account

### A — A frame cannot override perceptual partitions

The frame schema in `06-*` lets a frame request slot fillers; it
does not let a frame *re-partition* the inputs that came from
`perceive`. If `perceive` reports five blocks, no frame can say
"treat the top one as a body and the bottom four as two supports."
The dashed line of Figure 13-2 is precisely that re-partition, and
it has no operational form.

### B — No layered description with explicit demotion

Summary tiers add layers above detail, but the detail is not
*faded* in the runtime sense — it is simply unaddressed by the
brief. There is no flag on a signal that says "still present, but
not the description in force." Critics and censors see only the
current layer, with no machinery for asking "what did the lower
description say?"

### C — Reformulation is not reversible by design

Once a manifest or frame is changed and committed, the previous
description survives only in git history. There is no runtime
toggle that says "show me the same workspace under the older
vocabulary." Point 7 (reversibility) is honoured by git but not by
the runtime.

### D — A single overlay across heterogeneous instances has no representation

Figure 13-2 shows one template draped over four very different
arches at once. The plan has no object that represents "this
template, applied to this set of instances, holds." The closest
candidate is a K-line, but K-lines (per `06-*`) reactivate mental
states, not categorical equivalences. There is no
`equivalence.<name>` record.

### E — Slot expectations cannot be lifted from instance data

A frame's expected fillers are written by hand into the manifest.
The figure shows the opposite move: looking at four instances and
*inducing* that they all admit `body + supports`. Nothing in
[`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md)
collects "templates that fit recent runs" as candidate frames.

### F — Reformulation is treated as a code change, not a run-time sevent

In the plan, replacing the vocabulary requires editing manifests
and going through `frame.self-modification` and the
`differentiation-broker`. In the figure, reformulation is what
happens during perception, not during maintenance. There is no
phase between `perceive` and `activate` where the runtime is
allowed to try a different description of the same input.

## Implication for the plan (no changes proposed here)

Figure 13-2 is the figure that earns the word *reformulation* by
performing it. The implementation plan honours frames and tiered
summaries, but does not yet honour the perceptual-layer move at the
centre of this figure: imposing a partition the inputs do not carry
and treating it as the description in force.

The gaps above are recorded here as analysis, not as a change
request. Any move to close them — for example, a re-partitioning
hook between `perceive` and `activate`, an `equivalence.<name>`
record in [`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md),
or a "demoted layer" flag on signals — would touch
[`03-runtime-pipeline.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/03-runtime-pipeline.md),
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md),
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
and the identity scopes in `THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`,
and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
