# Figure 13-5 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [13-5.md](13-5.md) — *The same table from two
viewpoints.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 13-5 and checks
each one against the implementation plan.

---

## The ideas Figure 13-5 actually carries

From the caption and §13.2 *Boundaries*:

1. **The same object almost never presents the same shape twice.**
   Viewpoint shifts the silhouette continuously.
2. **Recognition requires constructive editing.** Without adding
   and removing edges, the head-on table and the perspective table
   would not collapse into one object.
3. **Invariance is achieved, not given.** The category *table* is
   the output of work, not a property visible in either image
   alone.
4. **Two descriptions are reconciled through a common template.**
   *Top + four legs* is what makes both pictures count as one
   table.
5. **The reconciliation is silent.** A perceiver normally never
   notices that they have done it; it is bookkeeping below
   awareness.
6. **The same template tolerates wide quantitative variation.**
   Foreshortened parallelogram and rectangle both satisfy *top*;
   four-leg and visible-two-leg both satisfy *legs*.
7. **The cost of skipping reconciliation is non-recognition.** If
   the editing did not happen, the two pictures would simply not
   refer to the same thing.

## What the implementation already absorbs

### 1 — Stable ids across runs

The id scheme in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md)
and `THE-SOCIETY-OF-REPO/02-protocols/01-identity.md` lets the
same agency be addressed across many runs. Two settlements about
the same agency can refer to it by a single name. Point 3
(invariance is achieved) is honoured for *agents*; the agency
remains itself across very different events.

### 2 — Memory and K-lines link earlier states to current ones

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
defines K-lines as records of useful prior activations.
`kline-retriever` brings back relevant past states; this is the
mechanism by which "the same situation under a different surface"
gets the same response. Point 4 (reconciliation through a common
template) has a partial home here.

### 3 — Signal-level deduplication

[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md)
allows multiple `perceive` signals to point at the same upstream
artefact via blob references. The plan can recognise that two
incoming events touch one object — a weak form of point 1.

## What the implementation does not yet take into account

### A — No viewpoint variable on inputs

`perceive` signals do not carry a `viewpoint:` or `surface-form:`
field. Two events that report the same underlying change from
different angles (e.g. the same PR as seen by `commit-watcher`
versus `issue-linker`) cannot be marked as "two views of one
thing." The plan can only deduplicate when blob references match
exactly.

### B — Constructive editing across views is absent

Reconciling Figure 13-5's two table images would require, in
plan terms, an agency whose job is "given several signals about
candidate-same-thing, build a unified description." No such
agency exists in the first-ship catalogues, and no critic asks
whether two recent perceptions are descriptions of one event.

### C — Invariance is not first-class

Identity in the plan is by stable id, which is correct for
*agents* but does not help for *situations*. There is no
`situation.<id>` scope in
`THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`, and no record
that ties several events together as "viewpoints on situation
X." The category-formation move of Figure 13-5 has no operational
counterpart.

### D — Tolerance bounds on slot-fill are not modelled

Point 6 — the same template absorbing wide quantitative variation
— would need slot fillers with declared tolerance (e.g. "this
slot accepts shapes within the *table-top* family"). Frame slots
in
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
are typed by intent, not by quantitative envelope, and there is
no validator that checks "is this filler close enough to count?"

### E — Reconciliation is overt, not silent

Whenever the plan deduplicates or merges, it does so through an
explicit agency that emits a signal and is auditable. This is
right for governance but means there is no analogue of point 5
(silent, sub-awareness reconciliation). Every join is visible —
which is good — but also means the runtime carries no concept of
"work the system did before any unit was consulted."

## Implication for the plan (no changes proposed here)

Figure 13-5 is the figure that earns the work of invariance by
showing how little perceptual common ground recognition actually
gets. The implementation plan honours invariance of *agents* by
id, but does not yet honour invariance of *situations* — the move
this figure is about.

The gaps above are recorded here as analysis, not as a change
request. Any move to close them — for example, a `situation.<id>`
scope, a `viewpoint:` field on `perceive` signals, or a
`agency.assembly.situation-reconciler` — would touch the manifest
schema in
[`05-agencies-critics-censors.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/05-agencies-critics-censors.md),
the signal schemas in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md),
and the identity scopes in
`THE-SOCIETY-OF-REPO/02-protocols/01-identity.md`, and so falls
under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
