# Figure 21-2 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [21-2.md](21-2.md) — *The Trans-frame.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 21-2 and checks each
one against the implementation plan.

---

## The ideas Figure 21-2 actually carries

Pulled from the figure and the surrounding text of §21.3
*Trans-frames*:

1. **A minimal frame skeleton.** Two endpoint slots — Origin and
   Destination — and one connecting arrow.
2. **The arrow is itself a labelled slot.** It can be filled with a
   Trajectory (a path) or a Difference (a change). The connector is
   not a hard-coded line; it is a third role.
3. **Same skeleton, many realms.** Spatial transport, information
   transmission, and ownership transfer share this shape. Schank's
   P-Trans, M-Trans, A-Trans collapse into one frame.
4. **Skeleton-reuse drives skill-reuse.** Because the shape is shared,
   the same chaining and reasoning machinery applies to all three
   realms. This is the load-bearing argument for *having* a Trans-frame
   at all.
5. **Origin and Destination are pronome roles, not pinned values.**
   The figure draws empty circles. What fills them is determined by
   the run, not by the frame.
6. **The arrow chooses between mode-of-travel and net-change.** The
   distinction between Trajectory and Difference is itself
   significant: one describes the journey, the other describes only
   the delta.

## What the implementation already absorbs

### 1 — A frame schema with named slots

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
provides a frame schema with a `slots:` map. A Trans-frame-shaped
frame *could* be expressed today as a frame with `origin`,
`destination`, and `trajectory_or_difference` slots. The schema does
not prevent it.

### 2 — Endpoint-shaped slots already exist for one concrete case

The state and memory model
([`08-state-and-memory.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/08-state-and-memory.md))
encodes the promotion path `state/` → `workspace/` → `memory/` —
itself an Origin/Destination shape, recorded as directory boundaries.
The settlement schema in
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md)
records the run's net change, which plays Difference's role for that
one domain.

### 3 — Frames are shared across stimuli of different kinds

The first-ship frame catalogue (`question`, `bug`, `feature`,
`code-change`, `security-sensitive`, `self-modification`, `novel`) is
the plan's instance of "one frame, many realms" — though here the
realms are stimulus kinds, not Trans realms.

## What the implementation does not yet take into account

### A — No Trans-frame as such

There is no `trans.frame.yml` in the first-ship catalogue, and no
generic "origin → destination" abstraction across the plan. Each
shipped frame is a *kind-of-stimulus* frame (bug, feature, etc.),
not a *shape-of-event* frame. The minimal three-slot skeleton that
Figure 21-2 promotes is not represented anywhere in the runtime
vocabulary.

### B — Slots cannot themselves be slot-of-slot

Idea 2 (the arrow is a labelled slot whose value is *which kind of
slot to use next*) requires a slot whose filler is itself a small
schema choice (Trajectory vs Difference). The frame schema's
`slots.<name>` has `required:` and `filled_by:` but no `oneof:` or
`schema_choice:` field that would let one slot say "I am either a
path-shaped slot or a delta-shaped slot." This is a real missing
expressiveness, not just a missing example.

### C — Cross-realm sharing is asserted, not enforced

The plan has frames for stimulus kinds, but it does not declare any
two frames to be "the same shape applied to a different realm." The
P-Trans / M-Trans / A-Trans equivalence in Minsky's text is exactly
that kind of declaration. The schema has no `extends:` or
`realises_shape:` field; a `physical-move.frame.yml` and a
`transfer-ownership.frame.yml` would be unrelated entries in the
catalogue today.

### D — Skill-reuse follows shape, but the plan reuses by id, not by shape

Idea 4 says that *because* the shape is shared, the *same skill*
applies. The plan reuses skills by naming critics and agencies
(`default_critics:`, `filled_by:`) rather than by matching shape.
A new frame must list its critics and fillers explicitly; nothing
says "any frame of Trans-shape automatically gets the chaining
critic." Without a shape-matching layer, the load-bearing argument
of Figure 21-2 — *one skeleton, one toolset* — is not yet operational.

### E — Trajectory vs Difference distinction is unnamed

The promotion path in `08-state-and-memory.md` is a Trajectory (it
names the steps). The settlement record is a Difference (it names
the delta). The plan currently treats both as ordinary outputs; it
does not distinguish "this artefact is the path we took" from "this
artefact is what changed." A Trans-frame implementation would put
both in the same slot under different sub-schemas. The plan has no
slot kind for this.

## Implication for the plan (no changes proposed here)

Figure 21-2 is the smallest possible reusable frame. The plan has a
frame schema rich enough to *contain* a Trans-frame but no Trans-frame
in its catalogue, no notion of *shape* shared across frames, and no
mechanism for shape-driven skill reuse.

Closing these gaps would mean either adding a `shape:` or `extends:`
field to the frame schema or introducing a parallel "skeleton"
registry. Either change touches
[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
and the SOR sources, and so falls under the "stop and ask" rules in
[`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
