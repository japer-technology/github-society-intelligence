# Figure 21-1 — Implementation analysis (SOR / FORGEJO-SOCIETY-IMPLEMENTATION)

**Source figure:** [21-1.md](21-1.md) — *Roles and concerns of an action.*
**Analysis target:** [`FORGEJO-SOCIETY-IMPLEMENTATION/`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/)
(plan-of-record for `.forgejo-society/` and
`.forgejo/workflows/forgejo-society.yaml`).

This file inventories the ideas carried by Figure 21-1 and checks each
one against the implementation plan. The aim is to mark, plainly, what
the plan already absorbs and what it does not yet name.

---

## The ideas Figure 21-1 actually carries

Pulled from the figure and the surrounding text of §21.2 *pronomes*:

1. **A fixed inventory of action-roles.** Origin, Action, Difference,
   Recipient, Motive, Trajectory, Object, Time, Destination, Actor,
   Cause, Method, Obstacle, Instrument, Vehicle, Place. The list is
   short, named, and recurs across very different accounts of an
   event.
2. **Role and filler are distinct things.** A slot ("Vehicle") is not
   the same kind of object as the value that fills it ("car").
3. **Roles travel together.** Once a situation is described as an
   action at all, the whole inventory is in play; missing values are
   not absent roles, they are unfilled slots.
4. **Two parallel columns of concerns.** The figure pairs each role
   with a *concern* — a question the role exists to answer ("What was
   changed?", "Who was affected?"). The slot carries the question
   shape as well as the answer shape.
5. **Grammatical universality is evidence of cognitive structure.**
   Every natural language already has dedicated machinery for these
   roles; the inventory is not invented for this figure, it is
   excavated.
6. **A scaffold for short tokens.** The list is the raw material the
   next two figures build on: pronomes are short symbols that stand
   for whatever currently fills each role.
7. **Decoupling of role from any particular agency.** The figure
   names roles without naming the agents that fill them. The same
   "Vehicle" slot will be filled by very different sub-agencies in
   different runs.

## What the implementation already absorbs

### 1 — Frames carry named, typed slots

[`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
defines a frame schema whose `slots:` field is exactly the "named
inventory of role-slots" idea. Each slot has a name, a `required:`
flag, and a `filled_by:` list of agent ids. The `code-change` frame
ships with `user_goal`, `relevant_files`, `proposed_patch`, `tests`,
`risks`, `final_user_response`; the `security-sensitive` frame adds
`permission_diff`, `revert_path`, `human_confirmation_state`. These
are the plan's equivalent of Figure 21-1's role column.

### 2 — Slot identity is separate from slot value

Frame slots are declared in YAML files under
`.forgejo-society/nemes/` and live independently of any particular
stimulus. Settlement instances (per
[`09-handoff-and-signal-schemas.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/09-handoff-and-signal-schemas.md))
record the values that fill those slots for one run. The "role vs
filler" distinction in idea 2 is operational.

### 3 — Failure conditions belong to the frame, not the filler

The frame's `failure_conditions:` and `default_critics:` fields name
the questions a frame must answer before settlement. This matches
the figure's "concern" column: each slot exists because there is a
question the run must not leave unanswered.

## What the implementation does not yet take into account

### A — The inventory itself is not first-class

Figure 21-1 names *sixteen* roles that recur across action-shaped
events. The first-ship frame catalogue defines slot names per frame
(`code-change` has six, `security-sensitive` adds three), but there
is no shared, cross-frame inventory of action-roles that a new frame
inherits from. Each frame redefines its own slot names. The plan has
no document equivalent to Figure 21-1's table — no
`nemes/role-inventory.yml` that says "Actor, Object, Method, etc. are
the canonical action-roles, and any new action frame should map onto
them."

### B — Slot-shaped roles are not separated from process-shaped slots

The `code-change` frame's slots mix roles (`relevant_files`,
`proposed_patch`) with phase outputs (`tests`, `final_user_response`).
Figure 21-1's discipline — *role-slots are about what the action is,
not about what the runtime is doing this minute* — is not enforced
schematically. The schema cannot say "this slot is a role; that slot
is a workflow artefact."

### C — The "concern" column has no analogue

In the figure, each slot is paired with a concern: a question the
slot exists to answer. The frame schema has `failure_conditions:` and
`default_critics:` at the frame level, but no per-slot `concern:` or
`question:` field. There is no way to declare, on the slot itself,
"this slot answers *why was it done?* and any critic asking that
question consults this slot."

### D — Grammatical-universality framing is absent

The figure derives its inventory from the observation that natural
languages already encode these roles. The implementation derives its
slots from `possibility-2.md` and from the operational shape of code
review. The plan does not claim — and currently has no mechanism to
test — that its slots correspond to anything found in language or in
other action-handling systems. This is by intent (the plan is
operational, not linguistic), but the consequence is that the figure's
*evidence* for choosing these particular roles does not transfer.

### E — No shared symbol space for role values

Idea 6 (short tokens that stand for slot contents) is the
*pronome* idea proper, and Figure 21-1 only sets it up. The plan has
polynemes that *excite* agencies by symbol, but no symbols that
*name a currently-bound slot value* and that any agency can read by
naming the role. The cross-figure gap is recorded properly in
[`21.5-sor.md`](21.5-sor.md) and
[`21.6-sor.md`](21.6-sor.md); here it appears as the missing other half
of the figure.

## Implication for the plan (no changes proposed here)

Figure 21-1 supplies the *catalogue* that the rest of chapter 21
elaborates. The plan honours the catalogue at the per-frame level —
frames have slots, slots are named, slots have fillers — but does not
yet honour the *cross-frame* level: there is no role inventory shared
across frames, and no per-slot concern.

The gaps above are recorded as analysis, not as a change request.
Adding a shared role inventory, a per-slot `concern:` field, or a
distinction between role-slots and process-slots would touch the
frame schema in [`06-frames-polynemes-klines.md`](../../../../../FORGEJO-SOCIETY-IMPLEMENTATION/06-frames-polynemes-klines.md)
and the SOR sources it cites, and so falls under the "stop and ask"
rules in [`AGENTS.md`](../../../../../AGENTS.md) §12 and
[`CLAUDE.md`](../../../../../CLAUDE.md).
